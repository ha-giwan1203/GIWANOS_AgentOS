# CODEX 작업지시 템플릿

> 목적: Claude가 Codex에 실행을 넘길 때 누락을 줄이기 위한 표준 양식.
> 비유: 작업지시서는 현장 작업 전 나눠주는 작업표처럼, 무엇을 만지고 어디까지 확인할지 미리 적어두는 표이다.

## 1. 목표
- 작업명:
- 최종 결과물:
- 성공 기준:

## 2. 작업 전용 하네스
> 명령이 들어오면 Codex는 바로 실행하지 말고, 먼저 이 5줄을 자체 설계한다.
> 비유: 작업 전용 하네스는 현장 작업 전 "무엇을 보면 통과인지" 정해 둔 검사표이다.
> 하네스 5필드가 빠지면 `doc_worklog.py start`와 `daily_doc_check.py`가 실패한다.

- 입력:
  - `<요청 원문, 기준 파일, 대상 데이터>`
- 작업범위:
  - `<바꿀 파일/폴더와 건드리지 않을 경계>`
- 성공기준:
  - `<무엇이면 완료인지>`
- 검증명령:
  - `<실행할 명령 또는 확인 방법>`
- 중단기준:
  - `<어떤 오류/불일치가 나오면 멈출지>`

### 큰 작업 추가 흐름
> 파일 2개 이상 수정, 코드/데이터/브라우저/시각산출물, 비가역 가능성이 있는 작업만 적용한다.
> 새 close-lite/full 같은 별도 구조를 만들지 말고 이 작업지시 안에 짧게 적는다.

- Plan:
  - `<목표, 제약, 잠금 파일, 먼저 읽을 문서>`
- Design:
  - `<바꿀 구조, 검증 기준, Claude 판단이 필요한 도메인 결정 여부>`
- Do:
  - `<최소 수정 순서>`
- Check:
  - `<자동검사, 실제 산출물 확인, git diff 확인>`

## 3. 잠금 파일
- 작업 시작 명령:
  - `python 90_공통기준/업무관리/doc_worklog.py start --task "<작업명>" --paths "<경로1>" "<경로2>" --harness-input "<입력>" --harness-scope "<작업범위>" --harness-success "<성공기준>" --harness-verify "<검증명령>" --harness-stop "<중단기준>"`
- 수정 대상:
  - `<파일 또는 폴더>`
- 건드리면 안 되는 파일:
  - `<파일 또는 폴더>`

## 5. 실행 지시
- 해야 할 일:
  - `<구체 작업>`
- 금지:
  - 사용자에게 파일 요청 금지
  - 원본 파일 직접 수정 금지, 단 사용자 또는 Claude 지시가 있으면 예외
  - 도메인 의사결정 금지
  - `goal mode`·`full access`·신규 플러그인 추가를 기본 운영으로 바로 확대 금지
  - destructive git 명령 금지

## 6. 검증
- 실행할 검증:
  - `<예: python ... --json>`
- 검증 원칙:
  - AI 완료 보고만으로 PASS 금지. 작업에 맞는 명령 결과·실제 산출물·브라우저 화면·엑셀/PDF 열림·git diff 중 최소 1개 이상을 확인한다.
- 작업 유형별 검증:
  - 브라우저 수집이면 조회 조건·건수·합계·예외를 확인한다.
  - PDF/이미지/HTML이면 열림·깨짐·잘림·겹침을 확인한다.
  - 코드 변경이면 문법검사 또는 관련 테스트를 실행한다.
- 확인할 결과:
  - `<PASS 조건>`
- 실패 시:
  - 원인과 재개 위치를 기록하고 멈춘다.
  - 마지막 줄에 `[NEEDS_CLARIFICATION] <필요한 결정>`을 남긴다.

## 7. 완료 처리
- 문서 갱신 명령:
  - `python 90_공통기준/업무관리/doc_worklog.py complete --task "<작업명>" --paths "<경로1>" "<경로2>" --handoff "[C] **<작업명> 완료**. 완료 내용, 검증 결과, 남은 예외를 1단락으로 기록." --status-title "<작업명> 완료"`
- 최종 검증:
  - `python 90_공통기준/업무관리/daily_doc_check.py --json`
- 리뷰:
  - 중요 변경은 `CODEX_리뷰루틴.md` 기준으로 Claude 검증 또는 `/codex:review` 대상 여부를 남긴다.
