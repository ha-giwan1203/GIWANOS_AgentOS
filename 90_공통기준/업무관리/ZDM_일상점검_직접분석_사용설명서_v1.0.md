# ZDM 일상점검 직접분석 사용설명서 v1.0

작성일: 2026-04-10
분석 방식: ZDM 직접 접속 후 전체 섹션 스캔 + 일상점검 화면/모달/API 분석
분석 상태: DRAFT

## 1. 목적

이 문서는 AI가 실제 ZDM 화면을 기준으로
`어떤 섹션이 있는지`, `일상점검에서 무엇을 조회/입력하는지`,
`어떤 데이터 구조로 저장되는지`를 빠르게 파악할 수 있도록 만든 직접 분석형 사용설명서다.

이번 버전은 문서 추정이 아니라
2026-04-10에 실제 `ax.samsong.com:34010`에 접속해 확인한 화면과
실시간 API 조회 결과를 기준으로 정리했다.

## 4. ZDM 공통 화면 구조

직접 확인한 ZDM의 공통 구조는 아래와 같다.

1. 상단 네비게이션에서 섹션 전환
2. 각 섹션은 `필터 -> 검색/초기화 -> 목록/통계 -> 모달 상세` 구조
3. 문서/점검/개선 화면은 버튼 클릭 시 Bootstrap 모달로 상세 입력
4. 목록 화면에서 `조회`, `입력`, `관리`, `수정`, `삭제` 액션이 한 화면에 함께 존재

MES/ERP와 비교했을 때 ZDM의 특징은 아래와 같다.

| 특징 | 설명 |
|------|------|
| SPA 구조 | iframe이 아니라 한 페이지에서 section 전환 |
| 문서 + 현장운영 혼합 | 조회만 하는 화면이 아니라 등록/수정/입력 기능이 함께 있음 |
| 이미지 결합 | 표준 이미지, 문제발생 이미지, 개선 전/후 사진 등 이미지 기반 운영이 많음 |
| API 기반 구조 | 목록/항목/기록이 REST API로 분리되어 있어 AI가 데이터 구조를 이해하기 좋음 |

## 6. 일상점검 API 구조

실제 페이지에서 호출 가능한 일상점검 API는 아래 4개 축으로 정리된다.

| 용도 | 메서드 | 경로 | 실제 역할 |
|------|--------|------|-----------|
| 점검표 목록 | GET | `/api/daily-inspection` | 점검표 마스터 목록 조회 |
| 점검항목 조회 | GET | `/api/daily-inspection/{masterId}/items` | 점검표별 항목 목록 조회 |
| 월별 점검기록 조회 | GET | `/api/daily-inspection/{masterId}/records?year_month=YYYY-MM` | 월 단위 기록 조회 |
| 점검기록 저장 | POST | `/api/daily-inspection/{masterId}/record` | 항목별 점검결과 저장 |

## 8. AI 질문 라우팅 기준

AI가 ZDM 일상점검 관련 질문을 받으면 아래처럼 연결하면 된다.

| 사용자 질문 유형 | 먼저 볼 기준 |
|------------------|--------------|
| 특정 라인의 점검표가 몇 개인지 | `/api/daily-inspection` 또는 일상점검 목록 |
| SP3M3 오늘/어제 점검이 완료됐는지 | `/records?year_month=YYYY-MM` 집계 |
| 특정 공정의 점검항목이 무엇인지 | `/items` 또는 `점검항목 관리(기준개정)` |
| 작업자가 어떤 항목을 점검하는지 | 입력 모달 표의 `점검항목/점검내용/점검자` |
| 문제발생 이력과 조치내역이 있는지 | 입력 모달 우측 `문제발생 조치이력` + `/records`의 `issue_desc`, `action_taken` |
| 표준 이미지를 확인하고 싶은지 | 점검표 master의 `image_path` 및 입력 모달 상단 이미지 영역 |

## 9. 이번 직접분석으로 확인된 핵심 포인트

1. ZDM은 단순 점검체크 화면이 아니라 `점검표 마스터`, `점검항목`, `점검기록`, `이미지`가 분리된 구조다.
2. `일상점검` 섹션은 실제 운영 액션이 강한 화면으로, 조회보다 입력/관리 비중이 높다.
3. SP3M3는 현재 `대원테크`, `19개 점검표`, `75개 항목` 구조로 운영된다.
4. 2026-04 기준 SP3M3는 `8일분 x 75건 = 600건`이 실제 저장돼 있었고 모두 `OK`였다.
5. 현장 질문 대응 시에는 목록 화면보다 `/records`와 `/items`를 함께 보는 편이 정확하다.

1. `공정진단`, `과거차 문제점관리`, `공정보증관리` 심층 분석
2. 점검결과 `NG` 사례가 있는 라인의 실제 조치이력 수집
3. 표준 이미지와 문제발생 이미지의 저장 규칙 정리
4. 작업자명/관리자명 입력 규칙과 권한 차이 확인
5. ZDM 문서(`관리계획서`, `작업표준서`)와 일상점검 간 연결 규칙 정리

## 11. 실무 메모

- ZDM은 브라우저 단에서 API를 직접 호출하므로, 화면 확인과 데이터 확인을 병행하기 좋다.
- `SP3M3-200-1`은 실제로 13개 항목이 있는 대표 점검표라 샘플 기준으로 쓰기 좋다.
- 현장 답변 자동화 관점에서는 `라인코드`, `공정번호`, `inspection_no`, `check_day` 네 축을 먼저 잡는 것이 가장 빠르다.
