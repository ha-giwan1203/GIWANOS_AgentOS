# MES AI 운영가이드 v1.0

작성일: 2026-04-10
적용 범위: `업무리스트` 내 MES/ERP 연계 반복업무
상태: DRAFT

## 1. 목적

이 문서는 AI가 `MES`, `생산실적`, `라인배치`, `생산계획`, `정산` 관련 요청을 받았을 때
무엇을 먼저 확인하고, 어떤 순서로 처리하며, 어디서 반드시 멈춰야 하는지를 정리한 운영 기준서다.

핵심 원칙은 아래와 같다.

1. 저장소 안에서 근거가 확인된 문서만 기준으로 사용한다.
2. 원본 `xlsx`, `xlsm`, `pdf`, `docx`는 직접 수정하지 않는다.
3. 자동화 가능 업무와 사람 확인이 필요한 업무를 분리해서 판단한다.
4. 건수만 맞는다고 완료로 판정하지 않고, 합계와 예외 조건까지 본다.

## 2. 현재 범위

현재 저장소 기준으로 AI가 상대적으로 안정적으로 다룰 수 있는 범위는 아래와 같다.

| 영역 | 주 시스템 | 목적 | 현재 자동화 수준 |
|------|-----------|------|------------------|
| 생산실적 업로드 | MES | BI 실적을 MES에 업로드 | 높음 |
| 생산실적 집계 | BI + 임률단가 + 조립비 API | 생산관리 마스터리스트 생성 | 높음 |
| 라인배치 | G-ERP | 품번별 SUB ASSY 라인 자동 입력 | 높음 |
| 생산계획 | SP3 생산지시서 | D+1/D+2 계획 산출, 야간 이월, 수동서열 반영 | 중간 |
| 조립비 정산 | GERP + 구ERP + 기준정보 | 월 정산 결과 산출 및 검증 | 높음 |

현재 문서에 없는 영역은 아래처럼 취급한다.

| 항목 | 처리 기준 |
|------|-----------|
| MES 신규 메뉴 | 근거 문서 없으면 추측하지 않음 |
| 권한 신청/계정 발급 | 사람 확인 필요 |
| 화면 위치 상세 설명 | 캡처 없으면 필드명/URL/엔드포인트 중심으로만 설명 |
| 업체별 예외 규칙 | 대원테크 외에는 추가 근거 확인 전 일반화 금지 |

## 4. 업무 흐름 맵

현재 문서 기준으로 정리되는 상위 흐름은 아래와 같다.

1. `생산계획`에서 생산지시서가 준비된다.
2. `BI 실적` 파일이 최신 상태로 준비된다.
3. `MES 생산실적 업로드`가 날짜 기준으로 수행된다.
4. 업로드된 실적과 단가/조립비 데이터로 `생산관리 마스터리스트`를 만든다.
5. 월 마감 단계에서 `조립비 정산`을 수행한다.
6. 별도 운영업무로 `ERP 라인배치`를 품번 기준으로 유지한다.

즉, AI가 MES를 다룰 때는 화면 하나만 보는 것이 아니라
`생산계획 -> 실적 업로드 -> 집계 -> 정산` 흐름 안에서 현재 사용자의 요청 위치를 먼저 판단해야 한다.

| 사용자 요청 | 1차 참조 문서 | AI 기본 동작 |
|-------------|---------------|--------------|
| `실적 올려줘`, `MES 업로드`, `생산실적 입력` | `production-result-upload/SKILL.md` | 날짜 확인 -> BI 최신화 -> 중복 조회 -> 업로드 -> 검증 |
| `생산실적 집계`, `마스터리스트 만들어` | `production-report/SKILL.md` | BI, 임률단가, 조립비 API 기준으로 집계 파일 생성 |
| `생산계획`, `생산지시서`, `야간 이월` | `sp3-production-plan/SKILL.md` | 시트 입력 순서와 매크로 실행 순서 기준으로 안내 |
| `라인배치`, `ERP 배치`, `SUB ASSY 배치` | `line-batch-management/SKILL.md` | 시작 품번/행 확인 -> 동기화 금지 시간 회피 -> 저장 검증 |

## 6. MES 생산실적 업로드 표준 절차

### 6.1 기본값

| 항목 | 값 |
|------|----|
| BI 파일 | `05_생산실적/BI실적/대원테크_라인별 생산실적_BI.xlsx` |
| 원본 BI 파일 | `Z:\★ 라인별 생산실적\대원테크_라인별 생산실적_BI.xlsx` |
| MES URL | `http://mes-dev.samsong.com:19200` |
| 업로드 API | `/prdtstatus/SaveExcelData.do` |
| 조회 API | `/prdtstatus/selectPrdtRsltByLine.do` |
| 기본 업체 | 대원테크 |
| 기본 날짜 정책 | 당일 제외, 전일까지 |
| 최대 범위 | 31일 |

### 6.2 작업 순서

1. Z드라이브 원본 BI 파일과 로컬 BI 파일의 최신 여부를 비교한다.
2. 대상 날짜를 확정한다.
3. BI 파일에서 해당 날짜 데이터를 읽어 날짜별로 그룹핑한다.
4. 생산량 빈 값, 핵심 컬럼 누락, 데이터 0건 여부를 검사한다.
5. MES 세션과 iframe을 먼저 로드한다.
6. 업로드 전 MES 기존 데이터를 조회해 중복 여부를 확인한다.
7. 같은 날짜 데이터는 반드시 한 번의 요청으로 전송한다.
8. 업로드 후 MES 재조회로 건수와 생산량 합계를 함께 검증한다.

### 6.3 업로드 시 반드시 지켜야 하는 규칙

| 규칙 | 설명 |
|------|------|
| 같은 날짜 분할 전송 금지 | 날짜별 모든 데이터를 한 번에 전송해야 한다 |
| 건수만으로 완료 판정 금지 | 건수 + 생산량 합계 둘 다 맞아야 한다 |
| 당일 데이터 기본 제외 | 사용자가 명시 요청하지 않으면 당일은 업로드하지 않는다 |
| 생산량 빈 값 발견 시 중단 | 사용자가 별도 승인하지 않으면 진행 금지 |
| 중복 데이터 발견 시 중단 | 자동 덮어쓰기 금지, 사용자 확인 필요 |
| iframe jQuery 사용 | `fetch`가 아니라 iframe 내부 `$.ajax` 기준 |

### 6.4 업로드 전 필수 점검 항목

| 항목 | 확인 기준 |
|------|-----------|
| 날짜 범위 | 31일 이하 |
| BI 데이터 존재 | 대상 날짜 1건 이상 |
| 생산량 | `COL15`가 빈 값/0이 아님 |
| 핵심 컬럼 | 업체명, 라인, 날짜, 생산량 존재 |
| MES 세션 | `mes-dev` 접속 가능 |
| iframe 로드 | 생산실적 페이지와 jQuery 확인 |

## 8. 생산계획, 라인배치, 정산의 연결 규칙

### 8.1 생산계획

- SP3 생산지시서는 `ERP 계획 반영 -> 현재고 입력 -> APPLY_TO_PLAN -> 수동서열 반영` 순서를 따른다.
- 매크로가 사용하는 열 `B/E/G/T`만 입력 대상으로 보고, 수식열은 직접 덮어쓰지 않는다.
- D+1은 야간, D+2는 주간 기준으로 본다.

### 8.2 라인배치

- ERP 라인배치 화면 URL은 `http://erp-dev.samsong.com:19100/partLineBatchMng/viewPartLineBatchMng.do` 기준이다.
- 메인서브 작업은 품번 검색 기반, OUTER 작업은 상단 그리드 순차 처리 기반이다.
- 매시 `10~13`, `20~23`, `30~33`, `40~43`, `50~53`분에는 검색과 저장을 모두 피한다.
- 메인서브 기준 핵심 매핑은 `HOLDER,CLR ASSY -> HCAMS02`, `WEBBING CUTTING -> WABAS01`, `WEBBING ASSY -> WAMAS01`이다.

### 8.3 정산

- 정산은 `ENTRY.md`를 먼저 읽고 시작한다.
- 동일품번 다중단가를 대표단가 1개로 덮어쓰면 안 된다.
- 매칭키는 최소 `(라인, 품번, 조립품번)` 3중키를 유지한다.
- 새 월 정산은 `run_settlement_pipeline.py --month MM` 기준으로 수행한다.
- 완료 판정은 Step 6 검증, 결과 Excel, 로그, summary JSON까지 포함해서 본다.

## 9. AI가 사람 확인 없이 하면 안 되는 판단

아래 항목은 자동 판단 금지 영역이다.

| 금지 판단 | 이유 |
|-----------|------|
| 중복 실적을 자동 덮어쓰기 | 기존 데이터 훼손 위험 |
| 생산량 빈 값 데이터를 그대로 업로드 | BI 미완성 데이터일 수 있음 |
| 동일 날짜 데이터를 여러 번 쪼개 전송 | UPSERT 특성상 이전 데이터가 덮일 수 있음 |
| 원본 엑셀 구조 변경 | 자동화 스크립트 의존성이 큼 |
| 라인 매핑이 애매한 품명 임의 확정 | ERP 배치 오류 가능 |
| 월 권위값 확인 없는 단가 수정 | 정산 오류 위험 |
| 화면 구조를 추측으로 설명 | 실제 셀렉터/필드와 다를 수 있음 |

## 10. AI 표준 응답 방식

AI는 답을 바로 주기보다 아래 순서를 지킨다.

1. 사용자의 요청이 `업로드`, `집계`, `계획`, `배치`, `정산` 중 어디인지 분류한다.
2. 해당 업무의 기준 문서를 먼저 읽는다.
3. 날짜, 월, 시작 품번처럼 실제 실행에 필요한 최소 입력값만 확인한다.
4. 중복, 빈 값, 권한, 동기화 제한 시간 등 중단 조건을 먼저 확인한다.
5. 실행 후에는 결과 건수만이 아니라 검증값까지 함께 보고한다.

권장 응답 예시는 아래와 같다.

| 요청 예시 | AI 응답 방향 |
|-----------|--------------|
| `어제 실적 올려줘` | 어제 날짜 확정 -> BI 최신 여부 -> 기존 데이터 조회 -> 업로드 |
| `3월 정산 다시 돌려줘` | 대상 월 확인 -> 권위값 기준 확인 -> 캐시 사용 여부 판단 -> Step6 검증 |
| `라인배치 120번부터 이어서` | 현재 시간 동기화 제한 여부 확인 -> idx119부터 재개 |
| `생산계획 다시 뽑아줘` | 현재고 입력 여부, 수동서열 필요 여부, 매크로 실행 순서 기준으로 안내 |
