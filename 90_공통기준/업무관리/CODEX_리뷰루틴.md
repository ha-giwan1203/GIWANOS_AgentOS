# CODEX 리뷰 루틴

> 목적: Codex가 실행한 결과를 Claude가 확인하고, 중요한 변경은 Codex 리뷰 기능으로 한 번 더 점검하는 운영 기준.
> 비유: 생산품을 출하하기 전에 외관검사와 치수검사를 하는 것처럼, AI 작업도 제출 전 검사 단계를 둔다.

## 1. 리뷰가 필요한 작업
- 도메인 자동화 코드 수정
- 정산, 라인배치, MES, D0 관련 변경
- 엑셀/문서/PDF 공식 산출물
- AGENTS.md, CLAUDE.md, hook, skill 변경
- git commit 또는 push 전 변경
- 사용자에게 바로 전달할 최종 보고서

## 2. 리뷰 생략 가능 작업
- 단순 파일 위치 찾기
- 읽기 전용 요약
- 임시 메모 작성
- 이미 검증된 스크립트의 동일 조건 반복 실행

## 3. 검증자 역할 분리
> 비유: Claude 검증은 현장 반장의 최종 승인이고, Codex Critic은 제품 하자를 찾는 별도 검사원이다.

| 검증자 | 주로 보는 것 | 결과 처리 |
|--------|--------------|-----------|
| Codex 자체 점검 | 바꾼 파일, 검증 명령, 작업 로그 누락 | 누락 수정 후 완료 보고 |
| Codex Critic | 숨은 버그, 경로 실존, 금지 위반, 테스트 누락 | FAIL이면 구현 단계로 되돌림 |
| Claude 검증 | 사용자 의도, 도메인 룰, git diff, 최종 전달 가능성 | 승인 후 사용자 보고·commit/push 허용 |

- 만든 AI와 검사 AI를 가능하면 분리한다.
- Codex Critic은 도메인 의사결정을 하지 않고, 코드·자동화·문서 정합성의 결함을 공격적으로 찾는다.
- Claude 검증은 최종 판정권이며, 특히 정산·라인배치·MES·D0 판단은 Claude가 맡는다.

## 4. Codex 작업 직후 자체 점검
- 변경 파일 목록 확인
- 의도하지 않은 파일 변경 확인
- 원본 파일 직접 수정 여부 확인
- 검증 명령 실행
- TASKS/HANDOFF/STATUS 갱신 여부 확인

## 5. Codex Critic 체크리스트
> 목적: 만든 쪽의 설명을 믿기 전에, 다른 눈으로 숨은 결함을 찾는다.
> 현재 수준: 자동 워커가 아니라 리뷰 단계에서 적용하는 체크리스트다. `/codex:review`나 Claude 검증 지시가 있을 때 이 기준으로 본다.

- 작업지시서의 목표·범위·금지사항과 실제 변경이 일치하는지 확인
- 참조한 파일 경로·스크립트·명령이 실제 존재하는지 확인
- 검증 증거가 있는지 확인 (`daily_doc_check`, 테스트, 산출물 열림, 브라우저 확인, git diff 등)
- 도메인 의사결정·원본 직접 수정·비가역 작업 같은 금지 위반이 없는지 확인
- 숨은 버그, 누락된 예외 처리, 실패를 PASS처럼 보고한 흔적이 없는지 확인
- 리뷰 결과가 FAIL이면 구현·커밋·푸시로 넘어가지 말고 원인과 재개 위치를 남긴다.

## 6. Claude 검증 기준
- 작업지시서 목표와 결과물이 맞는지 확인
- 도메인 룰 위반이 없는지 확인
- 사용자가 원하지 않은 비가역 작업이 없는지 확인
- 산출물 경로와 파일명이 맞는지 확인
- 실패·보류·예외가 숨겨지지 않았는지 확인

## 7. `/codex:review` 사용 기준
- commit 전 코드 변경이 있으면 사용 권장
- hook, skill, 자동화 스크립트 변경은 사용 강력 권장
- 정산/D0/MES처럼 실무 영향이 큰 변경은 사용 강력 권장
- 단순 문서 정리만 있으면 Claude 수동 검토로 충분

## 8. commit/push 게이트
> 비유: commit은 내부 보관함에 넣는 일이고, push는 외부 서버로 내보내는 일이다. push는 출하 전표처럼 별도 확인이 필요하다.

- commit 전에는 변경 파일 목록, 의도하지 않은 변경, 검증 명령, `TASKS/HANDOFF/STATUS` 갱신 여부를 확인한다.
- Claude 검증이 FAIL이면 commit/push로 넘어가지 않는다.
- `.git/index.lock` 권한 문제가 나면 먼저 lock 잔존 여부와 실행 중 git 작업을 확인하고, 일반 경로 재시도 후에도 실패할 때만 escalated commit을 1회 사용한다.
- push 전에는 Claude 검증 PASS와 사용자 push 발화가 모두 필요하다.
- push 전 사용자에게 원격 주소(`origin`), 브랜치, 커밋 해시, 외부 반출 고지를 제시한다.
- push가 차단되거나 실패하면 로컬 커밋 해시와 미푸시 상태를 보고하고, 우회·강제 push·원격 변경은 하지 않는다.
