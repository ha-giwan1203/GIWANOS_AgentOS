# 하네스 엔지니어링 운영 가이드

## 1. 개념 요약

AI에게 큰 작업을 시킬 때 발생하는 두 가지 문제를 구조로 해결한다.

| 문제 | 증상 | 해결책 |
|------|------|--------|
| 컨텍스트 소진 | 대화가 길어질수록 앞 내용 망각, 대충 마무리 | 컨텍스트 리셋 — 에이전트 교대 + 인수인계 메모 |
| 자기 평가 편향 | 자신이 만든 결과물을 후하게 평가 | 만드는 AI와 평가하는 AI 분리 |

## 2. 3인 체제 구조

```
[사용자 요청]
      ↓
[Planner] → 단순 요청을 상세 설계(10개 이상 항목)로 확장
      ↓
[Generator] → 설계대로 구현/문서/코드 작성
      ↓
[Evaluator] → 별도 컨텍스트에서 기준표로 판정
      ↓ (불합격 시 피드백 → Generator로 재전달, 최대 3회)
[완료 보고]
```

| 역할 | 입력 | 출력 | 금지 |
|------|------|------|------|
| Planner | 사용자 요청 1줄 | 상세 설계서 (항목 10개 이상) | 직접 구현 |
| Generator | 설계서 + 평가 기준표 | 구현 결과물 | 자기 평가 |
| Evaluator | 결과물 + 기준표 | PASS/CONDITIONAL/FAIL/BLOCKED + 피드백 | Generator와 동일 컨텍스트 |

## 3. 컨텍스트 리셋 규칙

- 에이전트 교대 시 HANDOFF.md에 **완료 내역 + 남은 예외 + 재개 위치만** 기록 (설계 설명 X)
- 다음 에이전트는 HANDOFF → STATUS → TASKS → CLAUDE.md 순서로 읽고 시작
- 장문 대화 누적 대신 문서 기반 재개 원칙 — 이전 대화 전체를 다시 먹이지 않는다

## 4. Evaluator 판정 규칙

### 판정 구조
| 판정 | 기준 |
|------|------|
| PASS | HARD GATE 통과 + 80점 이상 |
| CONDITIONAL | HARD GATE 통과 + 60~79점 |
| FAIL | HARD GATE 실패 또는 60점 미만 |
| BLOCKED | 외부 데이터 미존재, 동일 실패 2회 반복 |

### 재작업 규칙
- 불합격 시 Evaluator 피드백만 Generator에 전달 (이전 전체 대화 X)
- 최대 3회 재작업 허용
- 동일 실패 2회 반복 → BLOCKED 전환, HANDOFF.md에 원인·재개위치 기록

## 5. Known Exception 규칙

- STATUS.md에 등록된 기존 예외: 즉시 FAIL 아님, 감점 또는 INFO 처리
- 신규 예외 증가 (기존 등록값 초과): FAIL
- 기존 예외 해소 시: 해당 항목 PASS 가산 가능
- 판정 보고 시 Known Exception 목록과 실제 발생값을 함께 표기

## 6. 하네스 맵 — 현재 구현과 4단 하네스 매핑

> Agent = Model + Harness. 모델이 아니라 환경 설계가 경쟁력. (Anthropic)
> GPT 4단 하네스(Preflight/Response/Action/Regression)와 우리 실물의 1:1 매핑. (세션28 합의)

### 4단 하네스 매핑표

| 단계 | 역할 | 우리 실물 | 비고 |
|------|------|----------|------|
| **Preflight** | 작업 시작 전 점검 | `risk_profile_prompt.sh` (UserPromptSubmit) + `map_scope` 스킬 | 도메인 식별, 금지 차단, 체크리스트 |
| **Response** | 외부 응답 품질 판정 | `debate-analyzer` 에이전트 + `harness_gate.sh` (PreToolUse) | 채택/보류/버림 + 4조건 AND |
| **Action** | 비가역 행동 직전 게이트 | `commit_gate.sh` + `send_gate.sh` + `evidence_gate.sh` (PreToolUse) | commit/push/share 차단 |
| **Regression** | 회귀 검증 + 자기진단 | `smoke_test.sh` (105건) + `/self-audit` 메타 스킬 | 오프라인 + 온라인 분리 |

### 도메인별 하네스 적용 현황

| 도메인 | Preflight | Response | Action | Regression | Evaluator | Rubric |
|--------|-----------|----------|--------|------------|-----------|--------|
| 토론모드 | risk_profile | harness_gate | commit/send_gate | smoke_test | GPT (외부) | 채택/보류/버림 |
| 조립비 정산 | — | settlement-validator | commit_gate | smoke_test | subagent | HARD GATE + 100점 |
| 스킬 제작 | — | — | — | — | Evaluator (문서만) | 트리거30/오류25/실무25/문서20 |
| 영상분석 | — | A/B/C (비공식) | — | — | Claude 자체 | 9관점 + 등급 |
| 엑셀 수정 | — | — | evidence_gate | — | — | 없음 |
| 라인배치 | — | — | — | — | 예정 | 예정 |

### 보강 필요 영역 (빈 칸)
- 정산: Preflight(입력 파일 존재·월 확인) 추가 필요
- 영상분석: A/B/C를 공식 게이트로 승격 검토
- 엑셀 수정: Response(원본대조 결과 판정) 추가 필요
- 라인배치: 전 단계 신규 설계 필요

## 7. 스프린트 컨트랙트 템플릿

> 구현 전에 "무엇을 만들고 무엇으로 성공을 검증할지" 합의하는 최소 계약. (Anthropic)
> 비가역 영향이 크고 검증 기준이 명확한 작업에 우선 적용. (세션28 합의)

### 템플릿 (1페이지)

```markdown
# 스프린트 컨트랙트: [작업명]

## 범위
- 대상: [파일/도메인/파이프라인]
- 포함: [구체적 작업 항목]
- 제외: [이번에 하지 않는 것]

## 성공 기준
- [ ] [HARD GATE 1: 필수 조건]
- [ ] [HARD GATE 2: 필수 조건]
- [ ] [SCORE 기준: 항목별 가중치]

## 중단 기준
- 동일 실패 2회 반복 → BLOCKED
- 외부 데이터 미존재 → BLOCKED
- [도메인별 추가 중단 조건]

## 검증 증적
- 실행 로그: [경로]
- 결과 파일: [경로]
- diff/커밋: [SHA]

## 롤백
- 변경 전 백업: [경로 또는 커밋]
- 롤백 명령: [구체적 명령]
```

### 적용 대상
| 도메인 | 스프린트 컨트랙트 | 비고 |
|--------|-----------------|------|
| 조립비 정산 | **이미 비공식 운영** (CLAUDE.md HARD GATE + Rubric) | 템플릿 공식화 필요 |
| 라인배치 | **신규 적용 대상** | 설계 시 함께 작성 |
| 스킬 제작 | 하네스_스킬평가기준표.md가 역할 | 기존 유지 |
| 토론모드 | 불필요 (턴 단위, 비스프린트) | — |

## 8. Regression 편입 규칙

> 확인된 실패 사례를 regression 세트에 자동 편입하여 회귀를 방지한다. (세션28 합의)
> 전체 incident 자동 등록은 노이즈만 늘린다. P1/P2 확정 실패만 반자동 편입.

### 편입 흐름

```
incident_ledger.jsonl에서 P1/P2 확정 실패 감지
    ↓
smoke_test.sh에 테스트 케이스 초안 자동 생성 (파일 끝에 추가)
    ↓
사용자 확인 후 활성화 (주석 해제)
    ↓
다음 smoke_test 실행 시 자동 검증
```

### 편입 기준
| 등급 | 자동 편입 | 이유 |
|------|----------|------|
| P1 (서비스 영향) | 반자동 (초안 생성 → 승인) | 재발 시 실무 피해 |
| P2 (품질 저하) | 반자동 (초안 생성 → 승인) | 구조적 결함 가능 |
| P3 (개선 권고) | 편입 안 함 | 노이즈 방지 |
| resolved 24h+ | 편입 안 함 | 이미 해소 |

### 현재 상태
- smoke_test: 105건 (수동 관리)
- incident_ledger: P1/P2 자동 분류 가동 중
- **자동 편입 스크립트: 미구현** → 다음 작업 대상

## 9. 자기진화 루프

> "만든 다음이 진짜다" — 구축(30%) vs 운영·개선(70%). (장피엠)
> 진단 자동 + 수정 수동 승인으로 시작. 자동 수정은 2단계. (세션28 합의)

### 1단계 (현재 → 공식화)

```
매주 /self-audit 자동 실행 (scheduled-task)
    ↓
4축 진단: hooks/rules/skills/commands/agents/CLAUDE.md
    ↓
P1~P3 항목 검출 → 리포트 생성
    ↓
사용자에게 리포트 전달 (Slack/Telegram)
    ↓
사용자 승인 후 수정 실행
```

### 2단계 (진단 안정화 후)
- P3 자동 수정 허용 (문서 오타, 카운트 불일치 등 저위험)
- P1/P2는 여전히 수동 승인
- 수정 후 자동 smoke_test 실행

### 현재 상태 (세션134 갱신)
- /self-audit: 구현 완료. 수동 실행 + weekly 자동 실행. 세션134(2026-05-01)에 settings.local.json 1순위 표현 폐기 → `list_active_hooks.sh` SSoT 정렬 보강 완료(세션93 합의 적용)
- scheduled-task: `weekly-self-audit` 등록 가동 중
- **주간 자동 진단: 등록 완료** (매주 자동 실행 → 진단 리포트 생성. 첫 보강 후 작동은 다음 주 자동 실행에서 검증)
