## 3. skill-creator-merged.skill 현재 구조 파악

### 파일 위치
- `90_공통기준/스킬/skill-creator-merged.skill` (ZIP)
- 내부: `skill-creator-merged/SKILL.md`

### 실행 모드 (현재 v1)

| mode | 의미 | 깊이 |
|------|------|------|
| `draft` | 새 스킬 초안 생성 | Lite |
| `improve` | 기존 스킬 수정 | Lite |
| `eval` | 테스트/비교/채점 | Full |
| `optimize` | description/trigger 최적화 | Full |
| `package` | 최종 정리/배포 | Lite |
| `harness` | Planner→Generator→Evaluator 3단계 품질보증 | Full |

### harness 모드 트리거 조건
- "꼼꼼하게", "평가 포함해서", "하네스로", "3단계로" 등 품질보증 요청 시
- Full 모드: 설계 확정 → 구현 → 기준표 채점 → 피드백 루프

### 현재 한계 (1회차에서 확인된 사항)
- 하네스 평가 기준표 (`하네스_스킬평가기준표.md`)가 별도 파일로 존재하지만 skill-creator SKILL.md 내에 직접 임베드되어 있지 않음
- Known Exception 처리 로직이 SKILL.md 내에 명시되지 않음 → Evaluator가 매번 수동으로 STATUS.md를 확인해야 함
- 재작업 피드백 루프 최대 3회 규칙이 SKILL.md에 명시는 있으나 실행 절차가 구체적이지 않음

### 1회차 실패 항목 → 2회차 수정 대응표

| 1회차 실패/한계 | 2회차 수정 목표 | 방법 |
|--------------|--------------|------|
| 평가 기준표 미임베드 | SKILL.md 내 평가 기준 참조/요약 명시 | harness 섹션에 기준표 요약 또는 참조 경로 추가 |
| Known Exception 미명시 | 예외 처리 섹션 추가 | 예외 입력 경로 / 판정 반영 규칙 / FAIL vs WARNING 경계 명시 |
| 재작업 루프 불충분 | 루프 단계·중단 조건 표준화 | FAIL시 입력=Evaluator 지적, 처리=Generator 수정, 종료=PASS 또는 최대 3회 절차 명시 |

### 2회차 완료 판정 기준

| 기준 | 조건 |
|------|------|
| 기존 모드 구조 유지 | 6개 mode (draft/improve/eval/optimize/package/harness) 변경 없음 |
| 3인 체제 흐름 유지 | Planner→Generator→Evaluator 순서 변경 없음 |
| Evaluator 판정 | PASS (1회차 기준: 94점 이상) |
| 신규 치명 오류 | 없음 (신규 FAIL 원인 추가 없음) |
| 1회차 FAIL 원인 3건 | 모두 SKILL.md 내 문서 명문화 완료 |

## 4. 하네스 운영가이드 구조 파악

### 3인 체제 (출처: 하네스_운영가이드.md)
```
[사용자 요청]
      ↓
[Planner] → 단순 요청을 상세 설계(10개 이상 항목)로 확장
      ↓
[Generator] → 설계대로 구현/문서/코드 작성
      ↓
[Evaluator] → 별도 컨텍스트에서 기준표로 판정
      ↓ (불합격 시 피드백 → Generator 재전달, 최대 3회)
[완료 보고]
```

### 스킬 평가 기준표 (출처: 하네스_스킬평가기준표.md)
| 항목 | 가중치 | 합격 기준 |
|------|--------|---------|
| 트리거 정확도 | 30 | 의도한 상황에서만 발동, 오발동 없음 |
| 오류 처리 | 25 | 입력 누락·예외 상황 대응 명시 |
| 실무 적용성 | 25 | 즉시 사용 가능한 출력 형식 |
| 문서 완결성 | 20 | SKILL.md 필수 섹션 완비 |
