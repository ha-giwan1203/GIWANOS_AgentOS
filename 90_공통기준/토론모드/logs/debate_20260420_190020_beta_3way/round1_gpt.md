## 전문

쟁점 1 판정: 채택
- 근거 1: 6-2·6-4는 본론 생성이 아니라 "상대 답변에 대한 1줄 판정"이라, 누적 채팅 맥락보다 검증 대상 원문 전체 payload가 더 중요하고, 현행 SKILL도 교차 검증 시 원문 전체 동봉을 이미 강제하고 있습니다.
- 근거 2: 반대로 6-1·6-3 본론은 웹 UI 멀티턴 맥락 유지가 핵심이라, 같은 수준의 API 치환으로 보면 안 됩니다. 현재 규정도 3자 토론 도구를 GPT/Gemini 웹 UI 멀티턴으로 통일하고 있습니다.
- 근거 3: 그래서 "본론은 UI 유지, 단발 검증만 API"는 기술적으로 분리 가능한 안건이지만, 규정상으론 현행 [NEVER] API와 3자 토론 도구 통일 원칙을 건드리므로 규정 개정 전제의 예외 설계로 봐야 합니다.

쟁점 2 판정: 채택
- 근거 1: 현재 병목은 모델 추론 자체보다 탭 활성화, navigate 재호출, 입력, polling, 백그라운드 throttling 회피에 있고, 이 저장소도 그 비용을 줄이려고 세션84에 진입 캐시를 넣어 7회→3회까지 줄였습니다.
- 근거 2: 6-2·6-4는 짧은 검증 2회라서 UI 왕복의 고정비가 특히 아깝고, API는 이 고정비를 거의 제거하므로 라운드당 16~30초 절감 추정은 과장이 아닙니다.
- 근거 3: 다만 이 수치는 "정상 API 응답 + fallback 미발동" 기준치라서, 키 오류·rate limit·재시도까지 들어가면 체감 절감은 더 작아질 수 있습니다.

쟁점 3 판정: 채택
- 상대적 크기 순위: (b) 키 관리/비용/실패 fallback > (a) 모델 드리프트 > (c) 로그 보존성
- 근거: (b)가 기술 문제가 아니라 현재 [NEVER] API 규칙, 자격 증명 보관, 실패 시 UI로 자동 복귀할지 여부까지 엮여 있음. (a)는 본론 UI 동일 프로바이더 → API도 같은 프로바이더 쓰면 실질 영향 축소. (c)는 검증 결과만 JSON 로그 고정하면 해소.

쟁점 4 판정: 채택
- 근거: 안-A는 속도 이점에만 집중해 본론 UI 맥락과 단발 payload 설계를 묶지 않고, 안-B는 현상 유지지만 세션84에서 드러난 UI 병목 문제를 방치. 안-C는 원문 전체 payload + 병렬 API + 로그 보존을 묶을 수 있어 속도 이점과 통제 장치를 동시에 확보.

선호 안: **C**
- 이유: 본론(6-1·6-3)과 6-5는 웹 UI 멀티턴으로 남기고, 6-2·6-4만 원문 전체 payload + 병렬 API + 로그 고정으로 잘라내면 속도 이득은 가져가면서도 맥락 단절 위험을 가장 좁게 가둘 수 있음. 단, 현행 [NEVER] API 예외를 명시적으로 다시 합의해야만 성립.
