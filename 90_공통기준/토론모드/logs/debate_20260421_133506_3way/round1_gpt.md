# Round 1 — GPT 독립 의견

## D1~D5 검증

**D1: 대체로 동의지만 12개는 약간 많음**
Layer 1은 "고장 감지" 목적이라 정산·라인배치처럼 도메인별 깊은 정합성은 2차로 미루고, 처음은 8~10개로 시작한 뒤 실제 오탐 빈도로 늘리는 쪽이 안전.

**D2: 동의**
하드 임계값과 추세 임계값을 섞는 게 맞고, 최근 30일 p95는 좋지만 표본 부족 구간에서는 하한선과 최소 표본 수를 같이 둬야 함.

**D3: 조건부 동의**
정상 1줄, 이상 시 자동 펼침은 맞는데 "자동 펼침"이 토큰을 계속 잡아먹지 않게 WARN은 3줄 이하, CRITICAL만 상세로 제한.

**D4: 동의하되 외부 시스템은 "즉시 평가"를 네트워크 핑으로 이해하면 안 됨**
Notion처럼 read-only API가 이미 있는 대상만 짧은 timeout으로 확인하고, MES/ERP는 마지막 성공 시각·토큰 만료·최근 오류 로그 기반의 무접촉 판정이 우선.

**D5: 부분 반대**
1-click y/n도 운영 중엔 결국 누적 피로가 생기니, 자동 제안은 하되 "7일 3회 이상 반복 + 영향도 기준 충족"일 때만 승격 후보로 올리는 게 나음.

## Q1~Q5

**Q1**: D1은 8~10개 축소, D2·D3 대체로 찬성, D4는 무접촉 지표 우선, D5는 즉시 y/n보다 승격 조건 좁힘. 지금 안은 방향은 맞는데 Layer 1 치고 욕심이 섞임.

**Q2**: 빠진 건 settings drift와 Notion/MES/정산의 last_ok timestamp 존재 여부. 과한 건 hook 수≤36과 디스크·.bak 크기를 Layer 1에 바로 넣는 부분 — 전자는 정책이고 후자는 운영 청소라 고장 감지와 결이 다름.

**Q3**: SessionStart 1회 실행 + 결과 1세션 캐시 + 동일 세션 재계산 금지. invariant 평가 결과가 다시 incident를 대량 생성하지 않게 "요약 1건만 기록, 세부항목은 별도 JSON"으로 분리.

**Q4**: CLAUDE.md 조항만으로는 부족. 현재 실물 구조상 규칙 문서는 강제라기보다 지침이고, 활성 훅도 SessionStart·UserPromptSubmit·PreToolUse·PostToolUse·Stop까지만 있어 첫 응답 자체를 직접 보증하는 전용 응답 훅은 없음.

**Q5**: hook 수≤36은 B1에 넣지 말고 별도 의제로 분리. health invariant라기보다 구조 통제 정책이라 Layer 1 논의에 섞으면 고장 감지와 설계 규율이 뒤엉킴.

## 추가 위험 식별

**위험 1**: invariant 결과가 "정상/경고"를 넘어 자동 조치로 연결되기 시작하면 Layer 1이 아니라 Layer 3처럼 비대해짐. 처음엔 감지만 하고 조치는 절대 자동화하지 않는 선을 박아야.

**위험 2**: health summary를 첫 메시지에 강제하면 사용자가 단순 질문만 해도 매번 진단이 튀어나와 UX가 망가질 수 있음. "세션 첫 메시지"가 아니라 "프로젝트성 작업 시작 시 첫 메시지"로 범위를 좁힘.
