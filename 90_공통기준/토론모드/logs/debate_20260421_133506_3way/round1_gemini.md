# Round 1 — Gemini 독립 의견

## 1. D1~D5 각 결정 검증

**D1 (12개 항목): 이의**
12개는 초기부터 외부 의존성(ERP, API)이 섞여 1단계 도입 시 실패 지점이 너무 많음. 코어 시스템(Notion, Smoke, TASKS) 중심의 필수 5~6개로 시작해 D5 메커니즘 자체를 먼저 검증해야 함.

**D2 (혼합 임계값): 부분 동의**
p95 동적 산출은 초기 데이터 부족 시 오탐(False Positive) 유발. 첫 30일은 하드코딩 임계값만으로 운영하며 동적 산출값은 백그라운드 로깅으로 신뢰성만 검증하는 유예 기간 필요.

**D3 (조건부 보고): 동의**
인지 과부하 막으면서 문제 발생 시 즉각 가시성 강제하는 가장 합리적 UI.

**D4 (즉시 평가 + 5초 timeout): 이의**
세션 시작 시마다 외부 API(MES/ERP) 즉시 평가하면 5초 세션 진입 병목 누적. 외부 시스템 상태는 별도 비동기 크론 잡으로 갱신하고, health check는 결과(상태 파일)를 즉시 읽는 구조가 나음. 단, '파일 갱신 시각'을 함께 표시해 침묵(Silent failure) 방지.

**D5 (하이브리드 T2 1-click): 동의**
사용자 피로도 줄이면서 통제권과 변경 이력 명확히 남기는 최적 타협안.

## 2. 12개 invariant 후보 중 빠진 것·과한 것

**과한 것**: '외부 API 키 만료>30d' 및 '디스크·.bak 크기'는 인프라 모니터링 영역이므로 AgentOS의 1단계 자가 진단으로는 비대.

**빠진 것**: '미커밋 상태 지속 시간(Uncommitted changes age)'. Git 원본주의 원칙 수호 위해, 작업 후 커밋되지 않고 방치된 상태 탐지 항목이 최우선 필요.

## 3. 메타루프 위험 차단 구체 방안

평가 스크립트 자체에 무한 재시도를 막는 명시적 Depth/Retry 제한 하드코딩. 또한 진단 프로세스 자체를 OS 레벨 timeout 명령어로 감싸 실행하고, 시간 초과 시 절대 재평가하지 않고 즉시 'WARN(Timeout)' 상태로 반환하도록 격리.

## 4. 첫 메시지 health summary 의무 강제 메커니즘

**이의(별도 메커니즘 필요)**. CLAUDE.md 조항만으로는 프롬프트 컨텍스트 창이 밀리면 망각이 무조건 발생. 세션 시작 명령어 실행 시, 쉘 스크립트가 진단 결과를 터미널 최상단에 `[System Health]` 블록으로 먼저 출력하게 하여, Claude가 대화 시작 전 필수로 읽어들이도록 컨텍스트에 강제 주입.

## 5. invariant #12 (Subtraction Quota) 분리 여부

**별도 의제로 분리 요구**. #12는 단순한 '상태 측정(Self-Detection)'이 아니라 에이전트 행동을 제약하는 '자기 통제(Self-Limiting, Layer 2/4)' 철학. B1 단계에서 함께 섞어 논의하면 진단 시스템의 순수성이 훼손되고 합의만 지연.

## 추가 식별 위험: Alert Fatigue (경고 피로)

네트워크 순단 등 일시적 장애로 외부 시스템 진단이 실패할 때마다 WARN이 펼쳐지면, 사용자는 경고에 무감각해짐. 연속 N회(예: 3회) 실패 시에만 상태를 격상시키는 '디바운스(Debounce)' 로직이 invariants.yaml 설계에 반드시 포함.
