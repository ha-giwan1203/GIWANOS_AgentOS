# 의제 — Notion 동기화 범위 확장 (본문 정체 해소)

## 세션88 Notion 웹 직접 점검 결과 (실측)

**부모 페이지**
- 자동 갱신 시각 "2026-04-21 12:06" — 세션87 시점. 세션88 커밋 4건(14~15시) 미반영
- 하위 링크 제목 "(세션86 갱신 2026-04-21)" — 세션87/88 반영 안 됨

**STATUS 페이지 본문 (치명적)**
- "최종 업데이트: 2026-04-14 — 세션45" — 2주 정체
- "GitHub 동기화: 2026-04-14 / commit 18ae5cbe" — 2주 전 SHA 고정
- "훅 시스템 (23개 등록)" — 현재 44(raw)/36(활성)
- "smoke test 140/140" — 현재 215/215
- 완료 이력 블록: 세션46~88 전체 공백

## 근본 원인
세션86 3자 토론 합의(debate_20260421_102644_3way): `sync_from_finish()` wrapper는 "허위 이력 append 방지"가 목적이었고, 갱신 범위는 **"동기화: YYYY-MM-DD HH:MM KST" 1줄**로 제한됨. 본문 전반·페이지 제목·완료 이력은 갱신 범위 외.

결과: 사용자가 Notion만 보면 1~2주 완전 정체 상태로 보임. "TASKS.md > STATUS.md > HANDOFF.md > Notion" 원본 우선순위는 유지되지만, 보조 채널로서의 Notion UX가 심각히 저하.

## 쟁점 — 동기화 범위를 어디까지 확장할 것인가?

### A안: 범위 유지 + 정체 수용
- Git 원본주의 철저. Notion은 "동기화 시각"만 갱신
- 사용자에겐 "Notion은 신뢰 금지, Git만 보라" 명시
- 장점: 세션86 합의와 100% 일치, 추가 코드 없음
- 단점: 사용자 UX 지속 저하. "Notion 쓸 이유가 없다" 상태

### B안: 핵심 지표만 TASKS.md 최상단 복제
- STATUS 본문 중 "최종 업데이트 / GitHub SHA / 훅 수 / smoke / 완료 이력 최근 5세션" 필드만 자동 덮어쓰기
- TASKS.md 최상단 10~15줄을 Notion STATUS 전용 블록에 바인딩 (덮어쓰기, 누적 X — 허위 이력 방지 원칙 유지)
- 페이지 제목 자동 갱신은 옵션으로 분리 (Notion API `update_page` title 호출)
- 장점: 핵심 정체 해소, 허위 이력 방지 원칙과 충돌 없음
- 단점: STATUS 본문 구조 의존 — Notion 측 블록 ID 고정 필요

### C안: 페이지 제목 + 본문 집계 블록 자동 생성
- STATUS 본문을 매번 템플릿 렌더링 (`render_status.py` 신설)
- 완료 이력·집계·훅 수치 모두 동적 생성
- 장점: Notion만 봐도 완전 최신
- 단점: 세션86 "허위 이력 append 방지" 원칙과 충돌 소지 (템플릿 덮어쓰기 구조 설계 필요). 구현 비용 큼

### D안: Notion 자동화 폐기
- `notion_sync.py` 제거. 사용자는 Git/README.md만 참조
- CLAUDE.md "Git이 기준 원본" 원칙 엄격 적용
- 장점: 이중 관리 부담 완전 해소
- 단점: 사용자가 이미 Notion 활용 중 — UX 급격 저하. 세션86 합의 완전 반려

## Claude 독립 의견 (초안)

**선호: B안 변형 (B+) — 핵심 지표 덮어쓰기 + 페이지 제목 자동 갱신 옵션**

### 근거
1. **사용자 UX**: 사용자 질문 "웹으로 노션 직접가서 문제점 파악해바"는 "Notion도 쓸만해야 한다"는 암묵적 기대를 시사
2. **허위 이력 방지 원칙 준수**: 덮어쓰기 구조는 append 금지 원칙과 충돌 없음
3. **구현 난이도**: 중저. 기존 `notion_sync.py`에 `update_status_body()` 함수 추가로 처리 가능
4. **C안 대비 경제성**: 완료 이력 자동 생성은 복잡. 최근 5세션 블록만 복제로 충분

### 범위 제한 (허위 이력 방지 가드레일)
- 갱신 대상 블록은 Notion 측 고정 block_id로 식별 (TASKS.md 정규 템플릿 라인과 1:1 매핑)
- 덮어쓰기 단위는 "전체 블록 교체"만 허용, "라인 append" 금지
- 페이지 제목은 `notion_sync.py --update-title` 플래그로 옵션 분리 (기본 OFF)
- TASKS.md → Notion 단방향만 허용 (역방향 pull 금지)

### 불확실점 (양측 검증 요청 포인트)
1. Notion API `update_page` 로 대상 블록만 교체하는 안정적 경로 존재? (MCP `notion-update-page` 또는 block API)
2. 페이지 제목 자동 갱신이 "세션N 갱신" 포맷으로 매 커밋마다 바뀌면 Notion 검색·즐겨찾기에 악영향?
3. TASKS.md 템플릿이 세션별로 자유롭게 변하는데 "최상단 10줄 복제"가 안정적일까? 구조화된 프론트매터(YAML) 블록을 TASKS.md 상단에 신설해야 할까?
