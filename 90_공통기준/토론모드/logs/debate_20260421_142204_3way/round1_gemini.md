# Round 1 Gemini 응답 (B5)

**D1 이의**: 5개 전체는 관리 과부하. memory·command는 정원제보다 TTL(Time-to-Live) 자동 만료가 적합. 시스템 부하 직결되는 hook/skill/agent 3개 코어에 통제 집중.

**D3 이의**: 130% 버퍼는 '1 in 1 out' 긴장감 훼손. 100% 도달 시 즉시 1 in 1 out 강제 + 명시적 Bypass 플래그.

**D2 (Glide Path)**: 현재 초과 상태(hook 44)에서 즉시 36 강제 시 업무 마비. 현재 실측치를 초기 상한으로, 시스템 안정 시마다 매주 1~2개씩 깎는 점진적 하향.

**Q3 알고리즘**: 단순 mtime/빈도만으론 부족. **의존성(Dependency) 고립도** 필수. "최근 30일 호출 0 + 타 모듈(skill/agent) 참조 없음 + 보호 리스트 제외" 모두 충족 = 고아(Orphan) 훅 → 0순위 삭제 후보.

**Q4 면제**: 긴급 핫픽스 차단은 치명적. 커밋 메시지 [bypass-quota] 태그 시 1회성 통과. 단 기술 부채 기록 + 다음 정상 작업 시 '2개 삭제' 페널티 동반.

**Q5 보호리스트**: `90_공통기준/protected_assets.yaml` (또는 manifest.yaml) 별도 신설. OS 구동 필수 코어 시스템(Self-Detection 훅 등) 명시. D5 자동 제안 스크립트가 무조건 이 파일 최우선 읽어 후보군 원천 배제 — 아키텍처 단 강제.
