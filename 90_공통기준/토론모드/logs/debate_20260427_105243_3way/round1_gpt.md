### 의제0 — 메타 모순 해석

결론: 해석 C. P2-B만 조건부 토론, P2-C·P3-E는 보류.

근거는 간단합니다.

세션108의 핵심은 "시스템을 더 좋게 만들지 말고, 실무 산출물 모드로 전환하라"였습니다.
그런데 현재 P2-C와 P3-E는 명백히 운영 구조 확장입니다. 죽은 hook 운영 경로 부여, agent 진입 경로 명문화, 자동 호출 확대는 전부 "시스템 개선 모드 재진입"입니다.

반면 P2-B는 성격이 다릅니다.
evidence_missing 35건, auth_diag.req / identifier_ref.req 반복, GRACE 300초 초과는 단순 미관 문제가 아니라 정상 OAuth/D0 작업 중 차단 또는 incident 오염을 만들 수 있는 실제 운용 장애 후보입니다.
따라서 "근본 문제 마무리"라는 사용자 지시는 전체 시스템 개선 재개가 아니라, 실무 흐름을 계속 오염시키는 evidence 비대칭 1건만 최소 수정하라로 해석하는 게 일관됩니다.

약점은 있습니다.
P2-B도 결국 hook/evidence 영역 수정이라 시스템 개선으로 번질 수 있습니다. 그래서 수정 범위는 반드시 신규 hook 금지, settings 변경 금지, 정책 재설계 금지, 기존 evidence_mark_read 또는 risk_profile_prompt의 최소 보정으로 제한해야 합니다.

### 의제1 — P2-B evidence_gate auth_diag / identifier_ref ok 발급 절차

결론: 토론 진행 대상. 단, **Option B 우선**.

내 독자 견해는 Option B: evidence_mark_read 패턴 확장입니다.

근거는 이렇습니다.

문제의 본질은 req 발급 자체가 아니라 req는 자동으로 잘 나오는데, 정상 작업을 했다는 ok 증거가 안 찍히는 비대칭입니다.
즉, risk_profile_prompt가 "인증/식별 위험 있음"이라고 보는 것은 틀리지 않을 수 있습니다. 문제는 이후 정상 OAuth/D0 흐름이 증거로 인정되지 않는 겁니다.

그래서 Option C처럼 키워드를 좁히면 위험합니다.
chrome, login, oauth 같은 키워드를 줄이면 incident는 줄겠지만, 실제 인증 위험 작업까지 감지가 약해질 수 있습니다. 현장으로 치면 불량 줄이겠다고 검사 기준을 낮추는 짓입니다. 보기엔 조용해지지만 실제 위험은 남습니다.

Option A도 별로입니다.
D0/OAuth 스크립트 종료 시 직접 auth_diag.ok를 찍게 하면 업무 스크립트와 evidence 시스템이 결합됩니다. 나중에 스크립트가 늘어날 때마다 또 ok 발급 코드를 심어야 합니다. 이건 유지보수 지옥행입니다.

따라서 최소 수정은 Option B입니다.

방침은 이렇습니다:
- 신규 hook 추가 금지
- settings 변경 금지
- evidence_gate 정책 변경 금지
- evidence_mark_read.sh에서 정상 OAuth/D0 증거로 인정할 패턴만 좁게 추가
- 인정 조건은 "스크립트명만"이 아니라 "정상 완료 신호"까지 묶기 (예: erp_oauth_login.py 실행 + OAuth 200 또는 로그인 성공 로그)
- identifier_ref.ok도 무작정 발급하지 말고, 실제 기준문서/식별자 참조를 읽은 경우에만 인정

약점/리스크: 패턴을 넓히면 false ok, 즉 증거 없이 통과시키는 위험. 그래서 단순히 erp_oauth_login.py 이름만 보고 ok 찍는 방식은 반대. 성공 상태 문자열 또는 산출 로그까지 같이 봐야.

### 의제2 — P2-C 죽은 보조 hook 4종 운영 경로

결론: **보류**. 운영 경로 부여 금지.

근거: 대상 4종은 모두 시스템 유지보수 계층. 운영 경로에 넣는 순간 세션108의 "hook 추가 금지 / 시스템 개선 중단" 원칙을 깸. 특히 nightly는 세션77 합의가 있었어도, 지금 schtasks 등록까지 가면 자동 점검 체계가 다시 살아남. 실무 산출물 모드와 반대.

다만 폐기도 하지 않음. 폐기 역시 구조 변경. **동결 보류**가 맞음.

방침:
- settings 등록 금지
- schtasks 등록 금지
- 자동 호출 경로 부여 금지
- 삭제도 금지
- "보조 수동 도구 / 현재 운영 미사용" 상태로 둠
- 실무 작업을 실제로 막는 증거가 나오기 전까지 재논의 금지

약점/리스크: 세션77 Gemini 안전망 합의 약화. 그러나 지금 더 큰 리스크는 **안전망 부족이 아니라 안전망 과다로 업무가 막히는 것**.

### 의제3 — P3-E agents 7종 진입 경로 명문화 vs 폐기

결론: **보류**. 자동 호출 명문화 금지, 폐기도 금지.

근거: agents 7종의 진입 경로를 AGENTS_GUIDE에 추가하거나 자동 호출로 연결하는 것은 명백한 시스템 확장. 특히 self-audit-agent, critic-reviewer, debate-analyzer 같은 agent는 시스템이 다시 자기 자신을 평가하고 개선하는 루프를 강화할 가능성이 큼. 세션108에서 끊기로 한 바로 그 흐름.

그렇다고 폐기도 부적절. 폐기하면 또 "왜 지웠나 / 어디서 참조하나 / 보호대상인가 / 문서 갱신하라"가 따라옴. 이런 게 매번 볼일 다 보게 만든 원인.

방침:
- AGENTS_GUIDE 진입 테이블 추가 금지
- 자동 호출 위치 신설 금지
- agent 삭제 금지
- description 수정 금지
- 필요 시 수동 호출만 허용
- 실무 도메인에서 실제 산출물 검증에 필요한 경우에만 개별 사용

약점/리스크: agent가 존재만 하고 안 쓰이면 dead surface가 남음. 하지만 지금은 dead surface 정리보다 **시스템 안정 동결이 우선**. 죽은 도구를 정리하겠다고 또 시스템 구조를 뒤집는 게 더 위험.
