### 방안 A/B/C 독립 평가

**A (파일별 단일 스캔 러너로 재구성): 보류**
- 성능 이점은 확실하지만 Bash 중심의 현행 테스트 구조를 이질적인 언어로 통합하면, 향후 장애 발생 시 "어느 라인에서 왜 실패했는지" 추적하기 어려워지는 **유지보수성 하락(블랙박스화)** 우려 [일반 원칙]

**B (regression/capability + 파일 의존성 분할): 채택**
- 커밋 시점에 모든 기능을 검증할 필요 없이 핵심 회귀 테스트(regression)만 실행하는 것은 **CI/CD 파이프라인의 표준 최적화 기법**에 정확히 부합 [일반 원칙]

**C (섹션별/의존파일별 해시 캐시 + 의존성 맵): 채택**
- 훅 파일 하나가 바뀌었다고 전체를 무효화하는 **기존 30분 TTL의 무식한 구조를 버리고**, 변경된 범위만 정확히 타격하여 재실행하는 **가장 스마트하고 부작용 적은 접근** [논리 지적]

### GPT 우선순위 평가: 이의 (재배치 필요)
- GPT 순서: (1) 외부 통합 → (2) 해시 → (3) capability 분리
- **Gemini 재배치**: (B) 정책 분할 → (C) 섹션별 해시 → (c) grep 중복 통합 → A는 최후순위
- 이유: 코드를 뜯어고칠 필요 없이 실행 정책만 바꾸면 되는 **(3) capability 분리가 가장 빠른 병목 해소책(Low-hanging fruit)** [논리 지적]

### 병렬화(b) 보류 동의
- Windows MSYS2 환경에서 xargs -P나 & 백그라운드 포크를 남발하면, **컨텍스트 스위칭과 프로세스 생성 부하가 오히려 I/O 대기 시간을 압도**해버리는 안티패턴 [일반 원칙]

### GPT가 놓친 관점 — Test Pruning (Gemini 신규 지적)

> GPT는 167개의 테스트 케이스를 **'반드시 모두 실행해야 하는 성역'**으로 취급하고 있습니다. 하지만 167 PASS / 0 FAIL이 장기간 지속되었다면, 시스템의 안정성 때문이 아니라 **실효성 없는(거의 assert True 수준의) 과잉 검증 로직이 다수 포함되어 있을 가능성**을 의심해야 합니다. 외부 프로세스를 묶거나 캐싱하기 전에, **"이 테스트가 정말 필요한가?"를 따져 167개의 케이스 자체를 다이어트(Pruning)하는 과정이 선행**되어야 합니다 [논리 지적].
