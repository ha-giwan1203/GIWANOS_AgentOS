## GPT 응답 (전문)

### 즉시 적용

**8. context7 우선 규칙 명시**
라이브러리/API 문서 조회에 한정해서는 가장 먼저 넣을 만하다. Context7은 소스 저장소에서 최신 버전별 문서와 코드 예제를 끌어와 오래된 학습 데이터 문제를 줄이도록 설계돼 있고, Claude Code용 플러그인으로 안내돼 있다. 지금 저장소처럼 스킬·훅·MCP가 많으면 "라이브러리 문서 조회 시 WebSearch보다 Context7 우선, 일반 리서치는 WebSearch"로 선을 긋는 게 ROI가 높다.

**7. /statusline 적용**
거의 공짜에 가깝다. 상태 표시줄은 모델, 현재 디렉터리, 프로젝트 디렉터리, 비용 같은 세션 정보를 JSON으로 받아 한 줄로 표시할 수 있어서, 지금 저장소의 "경로/모델/컨텍스트 드리프트" 예방에 바로 먹힌다. 특히 main 직행 운영에서는 "지금 어느 경로에서 어떤 모델로 작업 중인지" 상시 노출이 사고 예방 효과가 크다.

**2. /doctor 세션 시작 점검 추가**
공식 문서도 claude doctor 실행을 권장. 현재 이미 session_start 계열 훅이 있으니, 별도 대형 기능보다 "session_start_restore → doctor-lite 체크"처럼 붙이는 편이 맞다. 다만 매 세션 풀 실행은 느릴 수 있으니, 시작 훅에는 경량 진단만 두고 본체 /doctor는 수동 또는 주 1회가 낫다.

**3. /rewind 한계 CLAUDE.md 명시**
공식 체크포인팅/베스트프랙티스 설명에서도 /rewind는 Claude가 만든 변경 되돌리기 안전장치지만, 외부 프로세스 변경까지 커버하지 않고 Git 대체재가 아니라고 못 박는다. 저장소는 Bash·외부 훅·MES/ERP 연동이 많아서 이 한계 문구를 명시하는 게 맞다.

**1. /insights 월 1회 실행**
이건 공식 명령 여부보다 커스텀 스킬/리포트로 보는 게 맞다. 지금 저장소엔 이미 TASKS/HANDOFF/incident/evidence 흐름이 있으니, 월 1회 패턴 리포트는 구현 부담이 낮고 관리 효율은 높다. /self-audit가 주간 구조 점검이라면 /insights는 "반복 실패 유형·훅 오탐·세션 패턴" 월간 요약으로 역할이 안 겹친다.

### 검증 필요

**4. /schedule 이관**
방향은 좋지만 바로 옮기면 사고난다. Claude Code 스케줄링은 /loop, Desktop, Cloud/Routines로 나뉘고, 클라우드 쪽은 컴퓨터가 꺼져도 돌지만 로컬 파일 접근은 안 되며 fresh clone 기준. Desktop은 로컬 파일 접근 가능하지만 앱이 켜져 있어야 함. GitHub/문서/리포트형 작업은 클라우드, MES/ZDM/사내망 의존 작업은 현행 로컬처럼 쪼개야 한다.

**9. skill-creator 플러그인 경로화**
커스텀 명령과 스킬은 같은 메커니즘, .claude/commands/*.md와 .claude/skills/*/SKILL.md 모두 슬래시 커맨드를 만든다. 장기적으로는 스킬 경로가 더 표준적이지만, 현재 저장소가 이미 .claude/commands/를 많이 쓰고 있으면 "즉시 승격"보다 신규 생성분부터 스킬 경로 우선, 기존 것은 점진 이관이 맞다.

### 보류

**5. /batch 활용**
기능 자체는 강력. /batch는 작업을 5~30개 독립 단위로 쪼개고, 각 단위를 격리된 git worktree 백그라운드 에이전트로 처리한 뒤 PR까지 연다. 문제는 너희 저장소가 PR 워크플로우를 버리고 main 직행이라는 점. 지금 구조에선 "도메인 STATUS 스키마 통일" 같은 대형 문서 리팩토링에 바로 태우기보다, 별도 실험 브랜치/워크트리에 박아두는 게 낫다.
