# Round 1 — Claude 독립 초안 (의제2 토큰 임계치 경고 스킬)

## 배경
- 세션68에서 Gemini가 제안한 안건
- TASKS.md/STATUS.md/HANDOFF.md 등 문서 비대화 → 세션 컨텍스트 오염 → 긴 세션 유지 시 컨텍스트 창 초과
- 사용자 피드백: 긴 세션에서 "세션 재시작 권고" 자동 알림 필요

## Claude 초안

### 측정 대상 (1안)
- TASKS.md 라인 수
- HANDOFF.md 라인 수
- MEMORY.md 라인 수 + 각 메모리 파일 개수
- 각 도메인 STATUS.md
- incident_ledger.jsonl 크기

### 임계치 초안
| 파일 | 경고 임계 | 차단 임계 |
|------|---------|---------|
| TASKS.md | 500줄 | 1000줄 |
| HANDOFF.md | 300줄 | 600줄 |
| MEMORY.md 인덱스 | 100줄 | 200줄 |
| 개별 메모리 총계 | 50개 | 100개 |
| incident_ledger.jsonl | 500KB | 1MB |

### 실행 지점 (후보)
- **A. SessionStart hook**: 세션 시작 시 1회 측정 → 임계 초과 시 stderr 경고
- **B. UserPromptSubmit hook**: 매 사용자 입력마다 측정 (오버헤드 큼, 부적합)
- **C. Stop hook**: 세션 종료 시 측정 → 다음 세션 대비 아카이브 권고
- **D. 독립 스킬 `/token-check`**: 사용자가 수동 호출

### Claude 권고: A + C 조합
- SessionStart에서 체크: 경고 출력 후 진행 (차단 아님)
- Stop에서 체크: 차단 임계 초과 시 "아카이브 필수" 플래그 설정 → 다음 세션 SessionStart에서 차단 모드로 전환

### 쟁점 (양측 제기)
1. Claude Code 내장 `/cost`·`/context`와 중복인가? 차별점은?
2. 임계치 수치가 현 저장소 규모에 적절한가? (TASKS.md 현재 약 200줄, HANDOFF.md 약 400줄)
3. 차단 임계 설정 필요한가? 경고만으로 충분한가?
4. 측정 방식: 라인 수 vs 바이트 vs 토큰 근사치 — 어느 것이 정확한가?
5. 아카이브 자동화 연동: 차단 임계 도달 시 `98_아카이브/` 자동 이동 여부?

### 독립 판단 (사전 명시)
- **Claude Code 내장 /cost와 역할 분리 필요** — 내장은 세션 토큰 소비, 이 스킬은 **저장소 문서 비대화** 측면
- **차단 임계는 반대** — 업무 중단 리스크. 경고 + 권고만
- **자동 아카이브 이동은 반대** — 중요 문서 임의 이동은 위험. 권고만 제시, 사용자 승인 후 실행
- **토큰 근사치 측정 반대** — 라인 수·바이트가 간단하고 결정적. 토큰 근사는 노이즈
