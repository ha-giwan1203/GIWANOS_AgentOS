# Round 1 — Claude 독자 답안 (선행 작성, 6-0)

세션: debate_20260428_201108_3way
**의제 (재정의 2026-04-28 사용자 지시)**: Opus 4.7을 사용 중인데 Sonnet 같은 추론 느낌이 드는 체감의 원인이 무엇인가. 단일 모델 라우팅 변경인가, 본 저장소 운영이 모델을 형식 기계로 길들인 결과인가, 아니면 둘의 합성인가.

> 이전 의제("사고+규칙 결합 장애의 외부 압력 지점")는 폐기. 그 의제는 체감의 결과론적 증거였을 뿐 본질이 아니었음.

## 결론 1줄

체감의 가장 유력한 원인은 본 저장소의 **컨텍스트 폭증 + 형식 강제**가 합성된 "운영 길들이기" 효과이며, 모델 라우팅 변경 가능성은 반증/입증 모두 어려우므로 별도 가설로 분리한다. 즉 모델은 Opus 그대로일 가능성이 높고, **출력이 Sonnet급으로 보이는 이유는 입력과 출력 형식이 Opus의 추론 자유도를 잠식하도록 설계되어 있기 때문**이다.

## 주장 1 — 컨텍스트 폭증 (가장 유력)

- **라벨**: 실증됨 (부분)
- **근거**:
  - 매 응답 입력에 누적 로드되는 항목: 글로벌 CLAUDE.md (한 페이지) + 프로젝트 CLAUDE.md (작업 모드 5종, R1~R5, 우선순위 사다리, 200+줄) + 데이터 처리 원칙 + cowork-rules + MEMORY.md 28+ 항목 + 토론모드 코어 (220줄) + 작업 모드 헤더 규칙 + 시스템 리마인더 + 사용 가능 스킬 100+ + 도메인 진입 표
  - 본 저장소 token_threshold hook이 이미 TASKS.md 1002줄을 STRONG 경고로 잡고 있음 (시스템이 토큰 폭증 자체는 인지)
  - 입력 토큰 측정은 매 응답 사용자에게 노출 안 됨 — Anthropic 측 정보. 다만 위 누적량은 정성적으로 명백
- **함의**: Opus라도 입력의 절대 다수가 메타 규정이면 본 사고에 할당 가능한 attention·연산 자원이 줄어든다. Sonnet급 출력은 Opus의 자원이 형식 처리에 묶인 결과일 수 있다.

## 주장 2 — 형식 강제로 인한 사고 자유도 침식

- **라벨**: 실증됨
- **근거**:
  - R1~R5 반증 질문형, A~E 모드 5종 판정, 채택/보류/버림 라벨 3종, Pre/Post Task 프로토콜, 모드 헤더 표기, 5라벨(실증됨/일반론/환경미스매치/과잉설계/구현경로미정), share-result 5단계 등 형식 슬롯이 응답 시작 전부터 구조를 결정
  - GPT i=77이 직접 짚은 사례: "원인 후보 1: 설정 / 2: 캐시 / 3: 경로" 식으로 채워서 통과되는 패턴 — 형식이 사고를 대체
  - 본 응답조차 모드 헤더 → 결론 1줄 → 주장 + 라벨 + 근거 → 권고 식 슬롯에 맞춰 작성 중. Claude 입장에서 자유 추론 경로가 거의 없음
- **함의**: Opus의 강점은 자유롭고 깊은 추론. 슬롯 채우기에 최적화된 출력은 Sonnet의 강점에 가깝다. 형식 강제가 강할수록 두 모델 차이가 줄어든다.

## 주장 3 — 메모리 누적 오염

- **라벨**: 일반론 (간접 증거)
- **근거**:
  - MEMORY.md 항목 28+ 중 feedback 16건이 행동 제약. 매 응답에 같은 제약 패턴이 주입되면 신규 자극에도 같은 반응 형성
  - feedback 메모리에 "결론부터", "짧게", "형식 맞춰", "검증 후 보고", "톤 통일" 등이 누적되어 응답 분포가 좁아짐
  - 직접 측정 어려움 — MEMORY 비활성화 비교 실험 본 저장소에 기록 없음
- **함의**: 같은 자극에 같은 답을 잘 하는 모델이 됨. 새로움이 줄어든 결과 "Sonnet 같다"는 체감.

## 주장 4 — 자기 메타화 비용

- **라벨**: 일반론
- **근거**:
  - 응답 첫 줄 모드 선언, 헤더 형식, 영향 범위 선언, 라벨링, R1~R5 적용, Pre/Post Task 등 메타 작업이 응답 토큰의 상당 부분 소비
  - 본 응답조차 토론 의제 자체가 메타이지만 추가로 형식 슬롯에 맞춰 작성 중
- **함의**: 본 사고 토큰이 메타 토큰에 잠식되면 출력 깊이가 얕아진다.

## 주장 5 — 사용자 톤 표준화 부작용

- **라벨**: 일반론
- **근거**:
  - "결론부터·짧게·형식" 등이 보수적·안전한 답을 유도. 창의/깊이 저하 가능
  - 메모리 `feedback_tone_rules.md` + 글로벌 응답 규칙
- **함의**: 정확하지만 평범한 답. Opus의 길고 깊은 추론 강점이 안 나옴.

## 주장 6 — 모델 라우팅 가설 (분리 평가)

- **라벨**: 검증 불가
- **근거**:
  - /fast 토글이 명시적으로 Opus 4.7 → Opus 4.6 다운그레이드라는 사실은 라우팅이 가능함을 시사
  - 그러나 백엔드 자동 라우팅 여부, 부하 시 모델 변경 여부 등은 사용자/Claude 측에서 검증 불가
  - claude-opus-4-7[1m] 모델 ID는 매 응답 시스템 컨텍스트에 표기되나 ID 표기와 실제 추론 모델 일치 검증 수단 없음
- **함의**: 입증·반증 모두 어려움. **이 가설을 묶어서 추론하면 다른 가설들의 책임이 가려질 위험**. 분리해서 별도 트랙으로 두는 것이 맞다.

## 핵심 후보 2개로 압축

- **A. 컨텍스트 폭증** (1+3+4 합성) — 입력의 비대화로 모델 사고 여력 침식
- **B. 형식 강제** (2+5 합성) — 출력 구조의 사전 결정으로 추론 자유도 침식

A와 B는 본 저장소의 동일한 진화 방향("Claude를 안전하고 예측 가능하게 만들기")의 두 면. 따로 측정 가능하나 함께 작동.

## 채택 권고 (Round 1 Claude 독자 결론)

1. **체감의 본질은 모델 다운그레이드가 아니라 운영 길들이기일 가능성이 높다.** 같은 Opus 4.7이라도 입력·출력이 형식 처리로 채워지면 Sonnet급 출력이 나온다.
2. **검증 우선순위**:
   - 1순위 (저비용): 글로벌+프로젝트 CLAUDE.md·MEMORY.md·시스템 리마인더 누적 토큰 실측
   - 2순위 (중비용): 형식 슬롯 강제 응답 vs 자유 응답 품질 A/B 비교 (동일 의제 두 번 답)
   - 3순위 (고비용): MEMORY 비활성화 세션 1회 비교
   - 라우팅 가설은 사용자/Claude 검증 불가능하므로 별도 트랙(Anthropic 문의 또는 외부 벤치)
3. **개입 후보 (조사 결과 따라)**:
   - 컨텍스트 폭증 입증 시 → 글로벌·도메인 문서 슬림화, MEMORY 정리, 도메인 진입 시 lazy-load
   - 형식 강제 입증 시 → 형식 강제 범위 축소(중요 작업만), 일반 작업은 자유 응답 허용
4. **무엇을 하지 말아야 하나**: 새 hook·새 모드·새 라벨 추가 금지. 본 진단의 결론이 "형식이 너무 많다"이면 형식을 더하는 해결책은 자가당착.

## 반대 안 예상 약점

- "느낌"은 측정 어려움. 사용자 체감 vs 객관 품질 측정 불일치 가능. 객관적 품질이 똑같은데 형식 때문에 느낌만 떨어졌을 수도 — 그래도 운영상 같은 문제.
- 컨텍스트 폭증 가설은 Anthropic 측 토큰 계산 미공개로 정량 입증 어려움.
- 형식 강제 축소는 안전성/예측성 저하와 트레이드오프. 본 저장소가 이 안전성을 의도적으로 만든 면이 있음.
- 라우팅 가설을 너무 빨리 분리하면 진짜 원인 놓칠 위험. 단 입증 가능한 다른 가설부터 검증 후 잔존이면 그때 라우팅 가설 재방문.

## 착수·완료·검증 조건

- **착수 조건**: 사용자 승인 후 1순위 검증부터 진행 (저비용 측정).
- **완료 조건**: 핵심 후보 A/B 중 우선 원인 식별 + 라우팅 가설은 별도 트랙으로 명시 + 개입 후보의 자가당착 회피.
- **검증 조건**: 합의안이 (1) 모델 다운그레이드/운영 길들이기 분리, (2) 측정 가능한 검증 우선순위, (3) "형식을 더하는 해결책 금지" 메타원칙을 모두 포함 + pass_ratio ≥ 2/3.

## 형식 함정 회피 메타원칙 (필수 명시)

본 토론의 결론이 "형식이 너무 많다"로 수렴할 가능성이 높음. 그렇다면 합의안 자체도 새 hook·새 슬롯·새 라벨을 추가하지 않아야 한다. 그렇지 않으면 동일한 자가당착에 빠진다. 합의안은 **빼는 방향**에 한정해야 한다.
