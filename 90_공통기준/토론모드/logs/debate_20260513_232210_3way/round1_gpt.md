# Round 1 — GPT 본론

> 모델: gpt-5-5-thinking
> 시점: 2026-05-13 23:22 KST

## 의제 1. UI 자동화가 진짜 병목인가?

판정: 조건부.

UI 자동화도 병목은 맞지만, 라운드 2~3분 전체를 잡아먹는 단독 주범은 아닐 가능성이 큽니다. 체감 비중은 모델 응답 생성이 대략 50~70%, 중지 버튼 감시와 텍스트 추출 대기가 15~30%, 개발자도구 왕복 호출과 탭 백그라운드 지연이 10~20% 정도로 봅니다. 특히 크롬은 숨겨진 탭의 반복 타이머를 1초 단위 또는 더 길게 묶어 처리할 수 있어, 현재처럼 버튼 상태를 반복 감시하는 방식은 느려질 수 있습니다.

콘솔 직접 호출로 바꾸면 버튼 클릭·감시·본문 추출 대기는 줄어듭니다. 다만 모델이 답을 생성하는 시간 자체는 그대로라서, 매 라운드 2~3분이 20초로 줄어드는 급가속은 어렵고, 현실적인 절감폭은 라운드당 20~60초 수준으로 보는 게 맞습니다. 긴 답변·추론형 답변이면 가속폭은 더 작아집니다.

## 의제 2. PoC 실행 가능성

판정: 조건부.

기술적으로는 같은 출처 페이지 안에서 콘솔 직접 호출을 하면 쿠키 같은 기본 인증 정보는 자동 포함될 가능성이 있습니다. 웹 표준상 브라우저 요청은 기본값이 같은 출처 인증 정보 포함이고, 응답 본문도 읽기 가능한 스트림으로 받아 누적 처리할 수 있습니다.

하지만 이것이 곧 안정 동작을 뜻하지는 않습니다. ChatGPT 쪽은 내부 호출에 쿠키 외 추가 토큰, 장치 식별값, 검증 헤더, 도전 응답 등이 붙을 수 있고, 이런 값은 단순 같은 출처 호출만으로 항상 자동 주입된다고 볼 수 없습니다. Gemini 쪽은 더 불리합니다. Google 계열 내부 호출은 요청 구조가 더 자주 바뀌고, 본문 형식·검증값·응답 형식이 웹 표준 텍스트 스트림보다 복잡할 가능성이 큽니다. 따라서 ChatGPT는 "기술 PoC 가능성 있음", Gemini는 "구현 부담 높음"입니다.

## 의제 3. ToS·계정 안전성

판정: 실패.

이 부분이 가장 큰 차단 사유입니다. OpenAI 공식 약관 계열 문서와 도움말은 자동·프로그램 방식의 데이터 또는 출력 추출, 제한 우회, 보호조치 우회를 금지 대상으로 설명합니다. 특히 ChatGPT 사용량 도움말도 무제한 이용이라도 자동 또는 프로그램 방식의 데이터 추출은 약관상 금지될 수 있다고 명시합니다.

Google도 서비스 약관에서 기계가 읽는 지시를 어기는 자동 수단 접근, 서비스나 기반 기술의 역공학성 접근을 금지 범주로 둡니다. Gemini 앱에는 Google 약관과 생성형 인공지능 금지 정책이 적용된다고 공식 도움말이 설명합니다.

같은 브라우저, 같은 쿠키, 같은 아이피에서 나가도 서비스 입장에서는 "사람이 화면에서 입력한 요청"과 "콘솔에서 내부 통신을 직접 호출한 요청"은 구분 가능성이 있습니다. 요청 간격을 5~10초로 낮추고 세션당 10회 미만으로 제한해도 안전하다고 판정할 근거는 없습니다. 이건 "빈도가 낮으니 괜찮다"가 아니라 "비공식 내부 경로를 직접 호출한다" 자체가 리스크입니다.

## 의제 4. UI fallback 회귀 보험 충분성

판정: 조건부.

제시한 4xx·5xx·시간초과·주소 변경·응답 구조 변화·세션 내 실패율 30%·주간 실패율 10%는 기술 회귀 보험으로는 방향이 맞습니다. 하지만 계정 안전성 리스크를 막는 보험은 아닙니다. 즉, "깨지면 UI로 돌아간다"는 운영 안정성에는 도움 되지만, "내부 호출을 해도 되는가" 문제를 해결하지 못합니다.

누락된 트리거는 최소 세 가지입니다. 첫째, 인증 도전이나 추가 검증 화면 감지 즉시 중단. 둘째, 응답은 왔지만 대화 기록에 저장되지 않는 경우 즉시 중단. 셋째, 공식 화면의 사용량 경고·임시 제한·보안 알림 발생 시 즉시 중단입니다. 임계값도 세션 내 30%는 너무 높습니다. 내부 호출형 PoC라면 실패율 기준보다 1회라도 인증·검증·제한 신호가 나오면 세션 전체 비활성화가 맞습니다.

## 합의 후보

폴링 단축 축소가 맞습니다. 콘솔 직접 호출 PoC는 기술적으로는 일부 가능성이 있지만, 약관·계정 리스크가 커서 폐기에 가깝고, 대신 현재 화면 기반 흐름을 유지하면서 중지 버튼 감시를 줄이고 개발자도구 이벤트 감지·탭 전면 유지·본문 추출 최소화로 가속하는 안을 채택하는 게 안전합니다.
