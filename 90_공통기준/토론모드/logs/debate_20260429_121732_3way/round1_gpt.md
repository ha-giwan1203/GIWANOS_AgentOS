# Round 1 — GPT 본론 (Step 6-1)

## verdict: 부분PASS

## item 1 (원인 분류 5종)
- **라벨**: 실증됨 / **판정**: 동의
- **근거**: timeout·5xx·네트워크·OAuth·CDP는 재시도 후보가 맞고, xlsx 미존재·권한·마스터 불일치는 재시도 금지가 맞음
- **추가**: "이미 일부 등록됨/중복 의심"을 별도 **RETRY_BLOCK**으로 추가해야 함 (B 분류)

## item 2 (백오프 정책)
- **라벨**: 일반론 / **판정**: 동의
- **근거**: 즉시/5/15/30분, 누적 50분은 아침 업무 시작 전 복구 시도와 과도 반복 방지의 균형
- **대안**: 즉시/3/10/20분, 누적 35분도 가능하나 ERP 응답 지연 고려하면 50분 안이 더 안전

## item 3 (UNKNOWN 1회 vs 2회)
- **라벨**: 환경미스매치 / **판정**: 반대 (Claude 2회 권장에 반대)
- **근거**: UNKNOWN은 "분류기 실패"라 2회 재시도하면 원인 모르는 자동 등록을 반복하는 꼴. 1회 즉시 재시도까지만 허용하고 실패하면 Slack 알림 + 로그 30줄 + 수동 확인

## item 4 (dedupe 선행 + schtasks 강제 종료)
- **라벨**: 실증됨 / **판정**: 보류
- **근거**: dedupe 선행은 필수. schtasks /end는 ERP 통신 중간에 끊을 수 있음 (Microsoft 공식 문서 인용 — 실행 중 작업 인스턴스 종료 명령). 강제 종료는 **"Running 5분 대기 후에도 Phase 0/1/2처럼 등록 전 단계임이 로그로 확인될 때만"** 허용. Phase 3(업로드) 이후면 종료 금지 + 알림

## item 5 (lock Windows 호환)
- **라벨**: 구현경로미정 / **판정**: 동의
- **근거**: os.open + O_CREAT|O_EXCL 방식은 락 파일 생성 경합 방지에 적합. Python 문서는 플랫폼이 O_EXCL을 제대로 구현한다는 전제에서 race condition 없다고 설명. Windows 운영에서는 **stale lock 정리 기준, pid·시각 기록, 만료 시간**까지 같이 넣어야 실무 안정성

## 추가제안
- **있음 / 분류=B**
- **내용**: RETRY_BLOCK 추가 — dedupe 결과 기존 등록·부분 등록 의심 시 재실행 금지 + Slack 알림

## 종합 의견 (400자 이내)
방향은 맞음. 단, "성공할 때까지"를 문자 그대로 구현하면 ERP 중복 등록 사고. 자동 재실행은 재시도 가능한 원인에만 열고, 매 시도 전 dedupe를 필수. UNKNOWN은 1회만, schtasks /end는 등록 전 단계 확인 시에만 허용. **핵심은 재실행기가 아니라 원인 분류 + 중복 차단 + 수동 알림 장치.**
