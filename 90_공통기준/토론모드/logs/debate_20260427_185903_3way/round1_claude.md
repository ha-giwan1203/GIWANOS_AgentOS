# Round 1 — Claude 독자 답안 (선행)

> 의제 그룹: 모드 5종 분류의 완전성 (의제1) + 모드 경계의 모호성 (의제2)
> 작성 시점: GPT/Gemini 본론 수령 **이전** (NEVER 생략 — 종속 방지)

## 결론 1줄

5종 분류는 출발선으로 충분하나, **F. 메타작업(메모리/회고/세션관리)**을 별도 분리해야 모드 C 오염을 막을 수 있다. 경계 모호성은 "**우선순위 사다리(E > D > C > B > A > F)**"로 해결한다.

---

## 주장 3~5개 + 라벨 + 증거

### 주장 1. 5종은 "공백" 한 군데가 있다 — 메타작업 누락
- **라벨**: 일반론 + 구현경로미정
- **근거**: MEMORY.md 인덱스(피드백 30건+, 프로젝트 5건)·자가진단(`self-audit`)·HANDOFF 정리는 어느 모드에도 안 들어간다. 모드 C(시스템 수정)로 끌어가면 R1~R5 풀 절차 적용으로 과부하, 모드 A(실무)로 끌면 "원본 보호"가 무의미.
- **제안**: **F. 메타작업** = 메모리/회고/HANDOFF/TASKS 정리/세션 인수인계. 절차: "현재 상태 스냅샷 → 정리 액션 → diff 보고". R1~R5 미적용, 토론 미승격.

### 주장 2. A↔C 경계는 "산출물 수신자가 누구냐"로 갈라야 한다
- **라벨**: 실증됨 (세션115 d0-plan 사고)
- **근거**: 세션115는 "실무 산출물(MES 업로드)" 작업 중 스킬 자체 수정으로 흐름이 끌려가 사고가 났다. 산출물 수신자가 **현장(ERP/MES/사용자)**이면 A, **시스템 자체(.claude/, 90_공통기준/스킬/, hook)**면 C.
- **테스트 케이스**: "d0-plan 첨부 가드 추가" → C (수신자=스킬 자체). "오늘 D+0 야간 계획 올려줘" → A (수신자=MES).

### 주장 3. B↔E 경계는 "현재 차단 여부"라는 시간축 1개로 충분하다
- **라벨**: 일반론
- **근거**: B(분석)는 "왜?"가 동기, E(복구)는 "지금 멈춤 해제"가 동기. 시간축이 다르다. 단, 사용자 보고 상태는 종종 "지금 안 돼" 형태라 자동 E로 빠지기 쉽다 → **체크 1단계 필요**: "이걸 안 고치면 사용자가 다음 30분 안에 차단되는가?" Yes=E, No=B.
- **반례 처리**: "1주일째 hook이 가끔 깨진다" → E 아님(차단 아님), B(분석) 진입. "지금 d0-plan이 ERP 로그인 못 한다" → E.

### 주장 4. 우선순위 사다리로 충돌 케이스 해소
- **라벨**: 구현경로미정
- **근거**: 케이스 충돌은 항상 발생한다(예: A 진행 중 시스템 결함 발견). "모드 일시중단 → 보고 → 재선언" 절차가 있어도 어느 쪽이 우선인지 사다리가 없으면 결정 비용이 매번 든다.
- **사다리**: **E > D > C > B > A > F**.
  - E(복구) 발견 시 즉시 다른 모드 일시중단.
  - D(토론)는 사용자 명시 호출만 발동, 다른 모드보다 위.
  - C(시스템 수정)는 B(분석) 결과로만 진입.
  - A(실무)는 F(메타)보다 우선 — 현장 산출물 마감이 메모리 정리보다 급함.

### 주장 5. 모드 선언은 "Claude 1줄, 사용자는 자유"로 비대칭화
- **라벨**: 메타순환 위험
- **근거**: 모드 표기를 사용자에게도 강제하면 학습 부담·입력 마찰. Claude만 응답 헤더에 1줄("[A. 실무 산출물로 진행합니다, 다르면 알려주세요]")로 선언 → 사용자는 다르면 1단어 정정.
- **단서**: 1초 안에 답할 단순 질의("오늘 며칠?")는 헤더 생략. 휴리스틱: "도구 호출 없이 답할 수 있으면 생략".

---

## 반대 안 예상 약점

1. **모드 F 신설 = 분류 비대화**: 5→6으로 늘면 판정 비용 증가. **반박**: F는 트리거가 명확(메모리/HANDOFF/TASKS/회고)해서 판정 비용 낮음.
2. **사다리의 도그마화**: E>D>C>B>A>F 순서가 케이스에 안 맞을 때 강제 적용되면 오히려 사고. **반박**: 사다리는 기본값, 사용자 명시 지시는 항상 사다리 위에 있다고 명문화.
3. **수신자 기준 (A↔C)이 모호한 케이스**: "현장에 배포할 스킬 신규 작성"은 수신자가 시스템(.skill)이지만 실무 직결. **반박**: 트리거 시 "스킬 작성 자체"가 목적이면 C, "스킬을 써서 산출물 만들기"가 목적이면 A.
4. **헤더 비대칭화 = Claude 자의적 판정 위험**: Claude가 매번 추정하면 오판 누적. **반박**: 사용자 1단어 정정 비용이 매번 묻는 비용보다 낮음 (실증 필요).
