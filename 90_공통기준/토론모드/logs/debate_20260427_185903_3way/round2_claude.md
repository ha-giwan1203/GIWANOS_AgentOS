## 의제 3 — 자동 승격 차단의 위험과 보완책

### 주장 3-1. "감지 ON, 액션 OFF"가 정답
- **라벨**: 일반론
- **근거**: 자동 승격을 끄면 구조 결함이 묻힐 위험은 실재한다. 그러나 끄는 대상은 **"자동으로 토론모드 진입"**이지 **"구조 결함 감지"**가 아니다. 분리하면 둘 다 산다.
- **구체화**:
  - 기존 `feedback_structural_change_auto_three_way.md`의 B 분류 키워드(hook/settings/gate 실행흐름·판정 분기 변경 등)는 **감지기로만 살린다**.
  - 감지되면 응답 말미에 1줄: `[B 감지: 구조 변경 후보 — /debate-mode 호출 권장]`. 토론 자동 진입은 안 함.
  - 모드 B(분석) 결과 보고에서도 동일하게 "토론 권장" 라벨 허용 (Claude 초안 §4 시나리오3 완화책).

### 주장 3-2. 사용자 명시 호출만이 D 진입 조건
- **라벨**: 실증됨 (세션84·115 사고)
- **근거**: 자동 승격이 실무를 멈춘 사고가 누적. "구조 결함이 묻힐 위험" < "실무 흐름 차단 사고 누적".
- **방어선 1**: B 모드 결과 보고에 토론 권장 라벨이 강제로 들어감 → 사용자가 보고 호출 결정.
- **방어선 2**: 주 1회 self-audit이 7일 누적 감지 라벨을 모아 보고 → 일괄 토론 의제로 묶기 가능.

### 주장 3-3. 차단 회수 조건
- **라벨**: 구현경로미정
- **근거**: 1주일간 "감지됐는데 토론 안 했더니 사고난 케이스"가 2건 이상 누적되면 차단 정책 재검토. 현 상태 무기한 유지 금지.

## 의제 4 — R1~R5 형식주의 리스크와 억제 장치

### 주장 4-1. 항목별 200자 상한 + "해당 없음" 명시 허용
- **라벨**: 일반론
- **근거**: 5개 항목을 의무화하면 빈 항목을 억지로 채우는 형식주의가 생긴다. 상한과 "해당 없음" 허용을 둬서 **간결성과 정직성 둘 다 강제**.
- **구체화**:
  - R1 (의존성 체인): 200자, 또는 "단일 파일·외부 의존 없음"
  - R2 (동일 패턴 grep): grep 명령어 1줄 + 결과 카운트, 또는 "신규 패턴"
  - R3 (호출 순서·등급): 200자, 또는 "실행 흐름 영향 없음 (문서 변경)"
  - R4 (회귀 시나리오): 시나리오 1~3개, 또는 "회귀 가능성 없음 (이유 1줄)"
  - R5 (롤백): 명령어 1줄 (`git revert <SHA>`), 또는 "롤백 불요(이유)"

### 주장 4-2. 회고 트리거 — Plan 분량 vs 실제 변경 분량 불균형
- **라벨**: 메타순환 위험 (자체 감지)
- **근거**: Plan이 실제 변경의 5배 이상이면 형식주의 의심. 단순 비율로 측정 가능.
- **구현 부담 vs 효용**: 자동 측정은 과잉. 사람(사용자) 6개월 회고에서 정성 판단으로 충분.

## 의제 5 — 응답 헤더 [모드] 표기

### 주장 5-1. 트리거 기반 선택 표기 — "도구 호출 발생 시"만
- **라벨**: 일반론
- **근거**: 모든 응답에 표기는 노이즈, 전혀 표기 안 하면 모드 오판정 발견 못 함. 임계는 **"이번 응답에서 Edit/Write/Bash/Skill을 호출했는가"**.
- **규칙**:
  - 도구 호출 0건 → 헤더 생략 (단답·설명·질문)
  - 도구 호출 1건 이상 → 헤더 1줄 (`[A. 실무 산출물]` 형식)
  - 사용자가 헤더 표기를 명시 요청하면 모든 응답에 표기

### 주장 5-2. Claude 1줄, 사용자 자유 (비대칭)
- **라벨**: 일반론
- **근거**: 사용자 입력에 모드 강제는 학습 부담. Claude만 헤더 1줄, 사용자는 다르면 1단어 정정.

### 주장 5-3. 모드 전환 시점 명시
- **라벨**: 구현경로미정
- **근거**: 작업 중 모드 전환(A → C 등) 시 별도 1줄 ("[모드 전환: A → C, 사유: ...]"). 누적 통계로 모드 오판정 회고 가능.

## 의제 6 — 모드 E "현재 실무 차단" 정량 기준

### 주장 6-1. OR 조건 3개 — "30분 차단" + "이미 시작" + "외부 응답 부재"
- **라벨**: 일반론
- **근거**: 정성 기준만 두면 항상 주관적. 구체적 OR 조건이 있으면 자기검증 가능.
- **OR 조건**:
  1. **시간 임계**: 사용자가 다음 30분 안에 진행할 작업이 차단됨 (예: ERP 로그인 안 됨, MES 업로드 실패)
  2. **이미 시작된 작업**: 시작한 작업이 중간에 멈춤(품번 50/100 처리 후 hang)
  3. **외부 시스템 응답 부재**: ERP/MES/Z드라이브 등 외부 시스템이 timeout·5xx·미응답
- **하나라도 해당** → E 진입.

### 주장 6-2. E 진입 후 "최소 패치" 정량 기준
- **라벨**: 일반론
- **근거**: "최소"가 모호하면 시스템 개선 우회로화 위험.
- **규칙**:
  - 패치 라인 ≤ 20줄 (의도적 제약)
  - 신규 함수·hook·gate·settings 추가 금지 (기존 값 변경·timeout 상향 등만)
  - 변경 파일 ≤ 2개
  - 초과 시 자동으로 모드 C 전환 + Plan 작성

### 주장 6-3. E 종료 후 사후 분석 강제
- **라벨**: 실증됨 (세션115 d0-plan 사고 이후 가드 추가가 실증)
- **근거**: E만 하고 끝나면 재발. E 종료 시 무조건 "B 분석 필요/불필요" 1줄 선언, 필요하면 TASKS.md에 등록.

## 반대 안 예상 약점

1. **"감지 ON / 액션 OFF" 분리는 결국 사용자 부담 전가**: 감지 라벨이 매번 보여도 사용자가 안 호출하면 묻힘. **반박**: 묻히는 게 자동 승격보다 낫다. 실무 흐름이 우선.
2. **R1~R5 200자 상한이 자의적**: 오히려 깊은 분석을 잘라낼 위험. **반박**: 200자는 핵심 압축 강제, 상세는 plan 본문에 따로.
3. **헤더 트리거 기준 "도구 호출"은 우회 가능**: 도구 안 쓰는 위험 작업도 있을 수 있음. **반박**: 도구 안 쓰는 응답은 사실상 텍스트만 → 영향 없음.
4. **모드 E 임계 "30분"은 자의적**: 케이스 다양. **반박**: 30분은 기본값. 사용자 명시 우선("지금 당장 안 되면 큰일")으로 무력화 가능.
5. **E 패치 20줄 제한이 비현실적**: ERP 로그인 한 줄 수정도 컨텍스트 포함 20줄 초과. **반박**: 단일 함수 한 줄 수정은 20줄 미만, 그 이상은 정말 시스템 개선 영역.
