쟁점 A — 채택.
1회용 permissions 16건은 그대로 두면 계속 누적되는 쓰레기 데이터라 제거가 맞고, permissions_sanity.sh로 PID·URL·1회성 rm/sed·완전 중복을 자동 탐지해 경고와 ledger 기록을 남기는 방향도 타당하다. PreToolUse의 Bash 매처로 이런 검사를 거는 구조 자체는 Claude Code 훅 모델과 정합하다.
Claude API Docs

다만 재발 방지는 "경고만"으로 끝내지 말고, 같은 패턴이 3회 이상 반복되면 정리 작업을 강제 권고하는 2단계 운영 규칙까지 붙여야 실효성이 올라간다.

쟁점 B — 채택.
완전 동일 중복 2건은 무손실 제거가 맞다. 이건 설계 논쟁거리가 아니라 데이터 정리다.
중복 제거를 permissions_sanity.sh의 기본 검사 항목으로 포함시키면, 이후 세션에서 또 같은 실수가 생겨도 바로 잡을 수 있다.

쟁점 C — 채택.
ls/wc/find/grep/cat/head/tail/echo 같은 Bash 허용을 전부 걷어내는 건 과하다. 실제 작업에서는 전용 도구가 우선이어도 Bash fallback이 필요한 순간이 있고, 특히 구조 점검·간단 집계·직렬 파이프라인에서는 여전히 쓸모가 있다.
대신 Claude 초안대로 루트 CLAUDE.md에 "전용 도구 우선, Bash는 fallback" 원칙을 명문화하는 게 핵심이다. 문서화가 붙으면 중첩 자체는 리스크보다 편의가 크다.

쟁점 D — 채택.
block_dangerous → commit_gate → debate_verify처럼 순서 의존성이 있는 상태에서 바로 통합부터 하면 오히려 사고 난다. 먼저 hook_common.sh timing 래퍼로 1주일 실측하고, 실제 병목이 어디인지 본 뒤 통합 여부를 결정하는 게 맞다. 훅은 matcher별로 순차 실행되는 구조라, 체감 오버헤드를 숫자로 본 다음 손대는 편이 안전하다.
Claude API Docs

즉시 통합은 "깔끔해 보이는 정리"일 뿐이고, 지금은 근거 없이 구조를 바꾸는 과잉행동에 가까운 것이다.

쟁점 F — 채택.
permissions = 도구 호출 허용/차단, hook = 호출 시점의 맥락 조건 검사라는 경계는 반드시 문서화해야 한다. 지금 문제의 절반은 기술보다 역할 경계가 묵시적이라는 데서 생긴다.
특히 신규 훅이나 permissions 항목을 추가할 때 "이건 on/off 문제인지, 맥락 조건 문제인지"를 먼저 묻게 해야 중복 설계가 줄어든다.

쟁점 D의 '1주일 timing 수집 vs 즉시 통합'은 과잉설계가 아니다.
지금 단계에서 timing 수집은 최소한의 계측이다. 오히려 즉시 통합이야말로 근거 없이 구조를 바꾸는 과잉행동에 가깝다.
네 저장소는 Bash PreToolUse가 이미 깊게 얽혀 있으니, 병목 위치를 수치로 확인한 뒤 손대는 게 맞다.

쟁점 A의 '경고만' 정책은 당장은 충분하지만, 영구 정책으로는 약하다.
세션71 시점에선 차단까지 갈 필요 없다. 1회용 permissions는 운영 중 실수와 잔재가 섞인 영역이라, 지금은 제거 + 경고 + ledger 기록이면 충분하다.
다만 완전 동일 중복과 명백한 1회용 하드코딩 패턴이 반복되면, 그때는 차단까지 올릴 수 있다. 즉 "지금은 경고, 반복되면 일부 항목만 하드 페일"이 맞다.

쟁점 F의 의사결정 트리 4단계는 거의 충분하지만, 1단계가 하나 더 필요하다.
내가 보강하고 싶은 건 맨 앞의 질문이다.
"이 항목은 프로젝트 전역 규칙인가, 이번 세션/이번 작업 한정 예외인가?"
이 질문이 먼저 있어야 1회용 permissions 같은 찌꺼기가 상시 정책 파일로 들어오는 걸 막을 수 있다.
그래서 완성형은 사실상 5단계에 가깝다:

1) 전역 규칙인가 일회성 예외인가
2) 도구 호출 허용 문제인가
3) 호출 시점 맥락 문제인가
4) 둘 다 필요한가
5) 기록/만료 정책이 필요한가

누락된 쟁점 G 후보 — .claude/settings.local.json을 기준 원본처럼 쓰는 현재 방식 자체.
공식 문서상 .claude/settings.local.json은 로컬 프로젝트 설정으로 설명되고, 커밋 대상 기본 원본은 아니다. 지금 감사는 항목 개수와 중복만 봤는데, 더 큰 쟁점은 "로컬 성격 설정 파일을 사실상 팀 공용 정책 저장소처럼 운영하고 있는 구조"다. 이건 permissions/hook 설계와 직접 연결되는 근본 이슈라 의제5에 추가할 가치가 크다.
Claude API Docs

이 쟁점을 분리하면 앞으로는 "팀 공용 정책은 .claude/settings.json 또는 기준 문서", "개인/세션성 허용은 settings.local.json"처럼 계층을 다시 나눌 수 있다.

합의 가능 — A·B·D·E·F는 채택 쪽으로 수렴 가능하고, C는 문서화 보완 전제 유지, G를 추가 쟁점으로 올리면 Round 1 설계가 훨씬 단단해진다.
