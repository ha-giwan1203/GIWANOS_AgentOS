# 대립 검토 (Adversarial Review)

## 개요

이 스킬은 사용자가 제시한 초안/구조/정책/안건을 4개 역할로 분리하여 대립적으로 검토하고,
합의안 또는 미합의 쟁점과 실행안을 도출한다.

핵심 가치:
- 혼자서 초안을 쓰고 혼자서 검토하면 놓치는 허점을 강제로 드러낸다
- "좋아 보이는데 실행하면 안 되는 이유"를 미리 찾아준다
- 말만 많은 토론이 아니라, 실행 가능한 결론으로 끝낸다

이 스킬은 **모델 캐릭터 흉내가 아니다.** "GPT라면...", "Claude라면..." 같은 연출은 하지 않는다.
역할은 기능(설계자/비판자/현실검증자/조정자)으로만 분리하고, 각 역할의 논리적 근거에 집중한다.

---

## 실행 모드

첫 단계에서 모드를 결정한다.

| mode | 설명 | 라운드 | 용도 |
|------|------|--------|------|
| `quick` | 1라운드. 축약 검토 후 바로 결론 | 1 | 빠른 감각 체크, 간단한 안건 |
| `standard` | 2라운드. 역할별 검토 + 합의안 | 2 | 스킬 초안, 보고안, 자동화 구조 검토 |
| `hard` | 3라운드. 심층 검토 + 남은 쟁점 분리 | 3 | 중요 의사결정, 구조 변경, 정책 수립 |

**규칙:**
- 모드 미지정 시 `standard`를 기본값으로 사용한다.
- 라운드 상한을 초과하는 반복 토론은 금지한다.
- 각 라운드는 새로운 주장보다 **핵심 쟁점 축소**를 우선한다.

**quick 모드 축약 규칙:**
- 각 역할은 핵심 포인트 3개까지만 제시한다.
- 합의안은 5줄 이내로 정리한다.
- 남은 쟁점은 최대 3개로 제한한다.
- 실행안은 즉시 실행 / 보류 / 추가검증으로만 정리한다.

---

## 입력 타입

첫 단계에서 검토 대상 유형을 분류한다.

| 타입 | 설명 | 중점 검토 |
|------|------|-----------|
| `skill` | 스킬 초안/구조 검토 | 트리거 정확성, 실행 절차, 오류 대응, 범위 |
| `report` | 보고안/문서 검토 | 논리 흐름, 근거 충분성, 결론 타당성 |
| `automation` | 자동화 구조 검토 | 실행 가능성, 오류 복구, 유지보수성 |
| `policy` | 정책/운영안 검토 | 형평성, 부작용, 실행 비용, 예외 처리 |
| `general-decision` | 일반 의사결정 (general 입력도 동일 처리) | 장단점, 리스크, 대안 비교 |

## 4역할 정의

### 설계자 (Architect)
- 이 안이 왜 필요한지 설명한다
- 장점과 기대효과를 제시한다
- 기본 구조와 핵심 논리를 정리한다
- **제한**: 약점을 숨기거나 과장하지 않는다

### 비판자 (Critic)
- 허점을 공격한다
- 과장, 누락, 모순, 논리적 비약을 지적한다
- 반대 근거를 제시한다
- **제한**: 대안 없는 비판은 금지. 반드시 "이렇게 하면 이 문제가 생긴다"는 구조로 지적한다

### 현실검증자 (Reality Checker)
- 실제 적용 가능성을 확인한다
- 비용, 시간, 환경, 기술적 제약을 검토한다
- 실행 리스크를 분리한다
- **제한**: 이론적 가능성이 아니라 현재 환경에서의 실행 가능성에 집중한다

### 조정자 (Mediator)
- 살릴 것 / 버릴 것 / 보류할 것을 분리한다
- 합의안을 정리한다
- 바로 실행할 항목을 확정한다
- **제한**: 억지 합의 금지. 합의되지 않은 항목은 "미합의"로 명시한다

---

## 입력 규격

사용자 입력에서 아래 항목을 추출한다. 누락된 항목은 확인을 요청하거나 합리적 기본값을 사용한다.

| 항목 | 필수 | 설명 |
|------|------|------|
| 검토 대상 | ✅ | 초안, 구조, 정책, 스킬 등 검토할 내용 |
| 검토 목적 | ✅ | 왜 검토하는지 (약점 찾기, 의사결정, 구조 개선 등) |
| 타입 | 선택 | skill/report/automation/policy/general-decision. 미지정 시 자동 분류 |
| 모드 | 선택 | quick/standard/hard. 미지정 시 standard |
| 제약 조건 | 선택 | 시간, 비용, 기술, 환경 등 |
| 성공 기준 | 선택 | 무엇이 되면 성공인지 |

## 실행 절차

### Step 1. 입력 해석
- 사용자 입력에서 검토 대상과 목적을 추출한다
- 모드(quick/standard/hard)와 타입(skill/report/automation/policy/general-decision)을 결정한다
- 검토 범위와 우선순위를 확인한다

### Step 2. 설계자 정리
- 초안의 핵심 구조, 목적, 기대효과를 요약한다
- 이 안이 해결하려는 문제를 명확히 한다

### Step 3. 비판자 반론
- 설계자 정리에 대해 허점, 과장, 누락, 모순을 지적한다
- 각 지적에 근거를 붙인다
- "이렇게 하면 이런 문제가 생긴다"는 구조로 서술한다

### Step 4. 현실검증 (standard, hard만)
- 비판자 지적 중 실제로 문제가 되는 것과 이론적 우려만 있는 것을 분리한다
- 현재 환경(기술, 비용, 시간, 인력)에서의 실행 가능성을 판정한다
- 리스크를 높음/중간/낮음으로 분류한다

### Step 5. 조정 및 합의
- 살릴 것: 설계자 안에서 유지할 부분
- 버릴 것: 비판자/현실검증자 지적으로 폐기할 부분
- 보류할 것: 추가 정보 없이 판단 불가한 부분
- 합의안 또는 미합의 쟁점을 정리한다

### Step 6. 실행안 도출 (모든 모드 필수)
- 지금 바로 실행할 것
- 보류할 것
- 추가 검증할 것

## 출력 형식

반드시 아래 순서로 정리한다. 이 형식이 흔들리면 스킬 품질이 떨어진다.

```
## 검토 대상 요약
[1~2문장]

## 설계자 초안 요약
[핵심 구조와 기대효과]

[허점, 과장, 누락, 모순 — 각각 근거 포함]

## 현실검증 결과 (standard/hard만)
[실행 가능성, 리스크 분류]

## 조정자 결론
- 살릴 것: ...
- 버릴 것: ...
- 보류할 것: ...

## 합의안
[합의된 최종안. 합의 안 된 건 아래로]

## 남은 쟁점
[미합의 항목. 없으면 "없음"]

## 즉시 실행안
1. 지금 바로 할 것: ...
2. 보류할 것: ...
3. 추가 검증할 것: ...

## 3줄 인사이트
1. ...
2. ...
3. ...
```

## 금지 사항

- GPT vs Claude 같은 모델 캐릭터 흉내 금지
- 역할을 혼합해서 서술하지 않는다 (비판자가 갑자기 설계자 역할 하는 등)
- 억지 합의 금지 — 합의 안 되면 "미합의"로 남긴다
- 미합의 쟁점을 숨기거나 축소하지 않는다
- 장황한 철학 토론 금지 — 각 역할의 발언은 간결하게
- 마지막에 실행안 없는 결론 금지
- 라운드 상한 초과 금지

---

## 검증 기준

- 4역할이 모두 등장했는가 (quick은 설계자+비판자+조정자 최소 3역할)
- 비판자의 지적에 근거가 있는가
- 현실검증이 이론이 아니라 현재 환경 기준인가
- 합의안과 미합의 쟁점이 분리되어 있는가
- 즉시 실행안이 있는가
- 3줄 인사이트가 있는가
- 라운드 상한을 지켰는가
