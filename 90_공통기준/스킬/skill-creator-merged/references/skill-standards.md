# 스킬 제작 표준 지침

이 문서는 skill-creator로 스킬을 만들거나 수정할 때 적용하는 품질 기준이다.
새 스킬 작성 시 이 문서를 참조하여 일관된 품질을 유지한다.

---

## 1. 네이밍 규칙

### 스킬명 (name)
- 영문 kebab-case 사용: `line-mapping-validator`, `adversarial-review`
- 동사+명사 또는 명사+명사 구조: `daily-report-writer`, `quality-data-checker`
- 3단어 이하 권장, 최대 4단어
- 금지: `my-skill`, `helper`, `temp`, `work-tool` 같은 모호한 이름

### 나쁜 예 → 좋은 예
| 나쁜 예 | 좋은 예 | 이유 |
|---------|---------|------|
| skill-1 | assembly-cost-settlement | 목적이 보임 |
| auto-helper | zdm-daily-inspection | 동작이 보임 |
| data-tool | line-mapping-validator | 대상+행동이 보임 |

---

## 2. Description 작성 공식

description은 스킬 트리거의 핵심이다. 아래 3파트 구조를 따른다.

```
파트1: 이 스킬이 무엇을 하는지 (1~2문장)
파트2: 언제 사용하는지 + 트리거 키워드 (한국어/영어 혼합)
파트3: 언제 사용하지 않는지 (과발동 방지)
```

### 작성 원칙
- 한국어 트리거 키워드 5개 이상 포함
- "반드시 이 스킬을 사용할 것" 문구로 언더트리거 방지
- "~에는 사용하지 않는다" 문구로 과발동 방지
- 업무 자동화 요청과 스킬 제작 요청을 구분할 수 있게 작성

### 예시
```yaml
description: >
  사용자가 재사용 가능한 Claude/Cowork 스킬 자체를 만들거나 수정하거나
  평가하거나 패키징해달라고 명시적으로 요청할 때만 사용한다.
  일반 자동화 요청, 문서 작성 요청, 데이터 처리 요청에는 사용하지 않는다.
  단, 해당 업무를 스킬로 전환하려는 의도가 분명한 경우에는 사용할 수 있다.
```

## 3. 출력 표준

### 보고형 스킬 (정산, 검증, 분석)
```
요약 → 상세표 → 개선안 → 3줄 인사이트
```

### 검토형 스킬 (adversarial-review 등)
```
요약 → 역할별 검토 → 합의안 → 남은 쟁점 → 실행안 → 3줄 인사이트
```

### 자동화형 스킬 (라인배치, ZDM 등)
```
사전확인 → 실행 → 결과 보고 → 오류 시 대응
```

### 공통 규칙
- 3줄 인사이트는 모든 분석/보고형 스킬에 필수
- 상세표는 문제 항목만 표시 (정상 항목은 건수 요약)
- 실행안이 있는 스킬은 반드시 "즉시 실행 / 보류 / 추가 검증"으로 끝낸다

## 4. 버전 관리 규칙

### 버전 표기
- `v1.0` — 최초 완성본
- `v1.1`, `v1.2` — 소폭 수정 (description 변경, 오타 수정, 규칙 추가)
- `v2.0` — 구조 변경 (패턴 추가, 역할 변경, mode 추가/삭제)

### 언제 버전을 올리는가
| 변경 유형 | 버전 | 예시 |
|-----------|------|------|
| 오타/문구 수정 | 마이너 (+0.1) | description 키워드 추가 |
| 규칙/블록 추가 | 마이너 (+0.1) | 오류 대응 시나리오 추가 |
| mode/역할/구조 변경 | 메이저 (+1.0) | quick→standard→hard 체계 도입 |
| 패턴 조합 변경 | 메이저 (+1.0) | P1+P4에서 P1+P4+P3으로 변경 |

### 구버전 처리
- 구버전은 `_구버전/` 폴더로 이동
- 파일명에 버전 표시: `스킬명_v1.0.skill`
- 최신본만 04_스킬/ 루트에 유지

## 5. 스킬 간 연동 규칙

### 연동 선언
스킬이 다른 스킬과 연결되는 경우, SKILL.md 하단에 연동 섹션을 둔다:

```markdown
## 기존 스킬 연동
- [선행 스킬] → [현재 스킬] → [후행 스킬]
- 예: line-batch-management → line-mapping-validator → (누락 품번 재배치)
```

### 연동 규칙
- 선행 스킬의 출력 형식과 후행 스킬의 입력 형식이 호환되어야 한다
- 연동되는 스킬끼리 같은 파일/폴더 경로를 참조하면 경로를 명시한다
- 스킬 A가 스킬 B를 자동 호출하지 않는다 — 사용자가 명시적으로 요청

## 6. 파일 구조 표준

### SKILL.md 단일 파일 기준
- 본문 500줄 이하 → SKILL.md 하나로 완결
- 매핑 테이블 50행 이상 → references/로 분리
- 스크립트 포함 → scripts/로 분리

### 분리 기준
| 항목 | 기준 | 위치 |
|------|------|------|
| 핵심 로직/절차 | 항상 | SKILL.md 본문 |
| 매핑 테이블 (50행+) | 길이 초과 시 | references/ |
| 도메인 가이드 | 재사용 가능 시 | references/ |
| 실행 스크립트 | 있으면 | scripts/ |
| 템플릿 파일 | 있으면 | assets/ |
| 평가용 에이전트 지시 | 있으면 | agents/ |

### 폴더 구조
```
skill-name/
├── SKILL.md (필수)
├── references/ (선택)
├── scripts/ (선택)
├── assets/ (선택)
└── agents/ (선택)
```

## 7. 금지 사항 (전 스킬 공통)

- 원본 파일 직접 수정 금지 (복사본에서 작업)
- 사용자 확인 없이 컬럼/시트를 임의 결정 금지
- 검증 기준 없는 완료 처리 금지
- 판정사유 없는 결과표 금지
- 수식 셀 임의 수정 금지 (data_only=False 유지)
- 접두어/코드 임의 제거 금지
