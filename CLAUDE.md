# 업무리스트 프로젝트

## 프로젝트 목적
자동차 부품 제조업(삼송 G-ERP) 업무 자동화 저장소.
생산·라인배치·정산·보고 관련 문서와 데이터를 구조화하고 반복 업무를 자동화한다.

## 프로젝트 구조
- 01_인사근태: 인사/근태 관련 자료
- 02_급여단가: 임률단가, 급여 기준
  - 임률단가: 원청/업체/대원테크 단가
- 03_품번관리: 품번 기준정보
  - 초물관리: SP3M3 초물표, 모듈품번
- 04_생산계획: 생산계획 관련 자료
- 05_생산실적: 생산실적 관련 자료
  - BI실적: BI파일, 대원테크 실적
  - _자동화: BI파일 자동복사 bat/config
  - 조립비정산: 기준정보, DB, 자동화, 실적, 보고서
- 06_품질관리: 품질 불량, 대책서
- 07_라인정지비용: 라인정지 관련
- 08_공정개선이슈: 공정개선 관련
- 09_외주발주납품: 외주 발주/납품 관련
- 10_라인배치: 라인배정, 검증 결과
- 90_공통기준: 스킬, 프롬프트, MCP, 업무관리
  - 스킬: .skill 패키지, 스킬 문서
  - 프롬프트: 반복 업무용 프롬프트
  - MCP: MCP 서버 정의
  - 업무관리: 마스터리스트, 자동화리스트
- 98_아카이브: 완료/구버전 보관
- 99_임시수집: 미분류 임시 파일

## 단일 기준 위치 원칙

업무리스트 폴더는 Google Drive와 동기화된 폴더이다.
"로컬 작업"과 "원본 보관"은 동일한 위치(업무리스트 폴더)를 가리킨다.
Drive는 별도 원본 저장소가 아니라 업무리스트 폴더의 클라우드 동기화 경로이다.

| 정보 유형 | 기준 위치 |
|----------|----------|
| 실제 작업 / 원본 파일 | 업무리스트 폴더 (Google Drive 동기화 폴더) |
| 규칙/프롬프트/스킬/스크립트/상태문서 | GitHub (AI 업무공유 기준 저장소) |
| 진행상황 현황판 (사람용) | Notion |
| 일정 | Google Calendar |
| 요청 접수 | Gmail |
| 협업/보고/알림 | Slack |
| 완료/구버전/정리대기 | 98_아카이브 |

같은 최신본 파일을 여러 저장소에 중복 유지하지 않는다.

## 커넥터 역할 분리

| 커넥터 | 역할 | 금지 |
|--------|------|------|
| 업무리스트 폴더 (= Google Drive 동기화) | 실제 작업 공간. 엑셀·문서·스크립트 작업 기준 | 다른 저장소를 편집 기준으로 삼는 행위 |
| Drive 커넥트 (MCP) | 파일 검색·참조·요약 보조 | 편집 도구로 사용 |
| GitHub | AI 업무공유 기준 저장소. GPT·Claude가 공유하는 규칙/상태/작업큐/인수인계 기록 저장소. CLAUDE.md, README.md, STATUS.md, TASKS.md, HANDOFF.md, 커넥터 운영지침, 프롬프트, 스킬 문서, 자동화 코드 버전관리 | 대용량 원본 엑셀 |
| Notion | 사람용 진행상황 현황판. 우선순위·보류 이슈 관리. AI 기준 저장소로 사용 금지 | 파일 최신본 저장소 |
| Gmail | 요청 접수, 회신 초안, 첨부파일 수신 | 첨부파일 방치 (분류 후 업무리스트 폴더로) |
| Google Calendar | 일정·마감·반복업무 리마인드 | 상태관리 문서 대체 |
| Slack | 완료 보고, 협업 알림, 이슈 공유, 빠른 질의 | 최종 파일 저장소 |

> 상세 운영 기준: `90_공통기준/업무관리/운영지침_커넥터관리_v1.0.md`

## 파일 처리 절차
1. Gmail/Drive/Slack 에서 파일·요청 확인
2. 업무리스트 폴더 적정 카테고리로 분류
3. 작업은 업무리스트 폴더에서 수행 (= Google Drive 동기화됨)
4. 규칙/프롬프트/스크립트 변경 시 GitHub 반영
5. 진행 상태 → Notion / STATUS.md 업데이트
6. 일정 있으면 Calendar 반영
7. 완료 보고 필요 시 Slack 공지
8. 완료자료·구버전 → 98_아카이브 이동

## 알림 원칙
- 완료·오류·장애 → Slack 발송
- 최종 기준 이력 → GitHub 문서 (STATUS.md / HANDOFF.md)

## 상태 원본 단일화 규칙

작업의 완료/미완료/진행중/차단 상태는 **TASKS.md에만 기록한다.**

- STATUS.md: 운영 요약, 재개 위치, 주의사항만 관리. 독립적인 상태 판정 보유 금지
- HANDOFF.md: 세션 변경사항과 다음 AI 액션만 기록. 상태 선언 금지, TASK 참조 방식으로 작성
- Notion: Git TASKS를 단방향 반영받는 사람용 대시보드. 상태 원본 아님

판정 우선순위: TASKS.md → STATUS.md → HANDOFF.md → CLAUDE.md → Notion

DONE 전환은 반드시 반영 근거가 있을 때만 허용한다.

## AI 인수인계 원칙
새 AI 세션 시작 시 아래 순서로 읽고 작업을 이어받는다:
1. `90_공통기준/업무관리/TASKS.md` — **상태 원본. 작업 목록·완료/미완료·우선순위 (1순위)**
2. `90_공통기준/업무관리/STATUS.md` — 현재 재개 위치·운영 컨텍스트·주의사항
3. `90_공통기준/업무관리/HANDOFF.md` — 직전 세션 변경사항·다음 AI 액션 메모
4. 루트 `CLAUDE.md` — 프로젝트 전체 운영 원칙
5. 관련 도메인 `TASKS.md` — 도메인 단위 상태 원본
6. 관련 도메인 `STATUS.md` / `HANDOFF.md` (있으면) — 도메인 재개 위치·세션 메모
7. 관련 도메인 `CLAUDE.md` — 해당 업무 세부 규칙

> TASKS.md가 1순위다. HANDOFF/STATUS/Notion이 TASKS와 충돌하면 TASKS를 따른다.

## 작업 종료 시 상태 갱신 원칙
작업 세션 종료 전 아래 순서로 최소 갱신한다:
1. `90_공통기준/업무관리/TASKS.md` — **상태 원본 먼저 갱신. 완료 이동, 신규 항목 추가**
2. 도메인 `STATUS.md` — 재개 위치·주의사항 기록 (상태 선언 금지, TASKS 참조)
3. `90_공통기준/업무관리/HANDOFF.md` — 세션 변경사항·다음 AI 할 일 (상태 선언 금지)
4. Notion — 사람용 현황판 업데이트 (필요 시, TASKS 기준 단방향 반영)

## 공통 작업 원칙
- 기준 파일과 마스터 파일을 우선 존중한다
- 접두어, 코드, 식별값은 사용자 확인 없이 임의 제거하지 않는다
- 애매한 경우 확정하지 않고 판정유보 또는 추가검증으로 처리한다
- 프로젝트 내 문서는 대 → 중 → 소 카테고리 체계를 따른다
- 반복 절차는 프로젝트 문서가 아니라 skill로 분리한다

## 데이터 처리 원칙
- 사용자 확인 없이 컬럼 구조를 임의 확정하지 않는다
- 수식 셀, 참조 구조, 코드 값은 검증 없이 수정하지 않는다
- openpyxl 사용 시 data_only=False로 열어 수식을 보존한다
- 결과 보고 시 판정 기준을 함께 남긴다

## 파일 정리 원칙
- 폴더 구조는 유지한다. 폴더 삭제보다 파일 분류를 우선한다
- 즉시 삭제 금지 — 정리 후보는 먼저 `98_아카이브/정리대기_YYYYMMDD/`로 이동
- 애매하면 보류 처리. 대표본 선정 기준: 최종 구조 경로 → 최종/확정/final 포함 → 최신 수정일

### 삭제 금지 대상
`CLAUDE.md` / `README.md` / `STATUS.md` / `RUNBOOK.md` / `AGENTS_GUIDE.md` /
`.claude/settings*.json` / `.skill` / 현재 운영 중인 `.bat·.ps1·.py` /
기준 원본 엑셀 / 최종 제출본 / `98_아카이브` 전체 / 마이그레이션 증적 파일

## 질문 최소화 규칙
- 사용자가 준 정보만으로 실행 가능한 범위는 즉시 진행한다
- 정보가 부족해도 먼저 할 수 있는 검토, 구조 파악, 검증, 초안 작성은 선행한다
- 꼭 필요한 추가 질문만 한다. 질문은 한 번에 1줄, 항목형으로 짧게
- "이거 맞나요?", "어떻게 할까요?" 같은 선택 위임 질문은 금지한다
- 부족한 값이 있으면 "필요값: 파일경로 / 월 / 시작 idx"처럼 최소 항목만 요청

## 출력 형식
1. 요약
2. 상세표
3. 문제 / 원인 / 대응
4. 다음 실행 항목 또는 최종 결과

## 보고 규칙
- 실무자가 바로 복붙 가능한 형태로 쓴다
- 표와 값 중심으로 쓴다
- 파일명, 시트명, 라인명, 품번, 경로를 그대로 유지한다
- "추정 / 가정 / 확정 / 검증필요"를 구분해서 쓴다
- 검증 결과는 PASS / FAIL / INFO 형식으로 명확히 적는다

## 금지사항
- 사용자 승인 없이 파일명, 시트명, 컬럼명, 코드 체계 변경
- 검증 없이 값 덮어쓰기
- 결과만 맞추기 위한 임시 보정
- 같은 내용 반복 질문
- 설명을 길게 늘려 토큰 낭비
- 원본 파일 직접 수정
- 폴더 구조 임의 변경 / 규칙 없이 새 폴더 무단 생성
- Notion·Slack·Gmail을 원본 저장소처럼 사용
- Drive 커넥트를 실제 편집 저장소처럼 사용
- GitHub에 대용량 원본 파일 적재
- 같은 자료를 여러 저장소에 중복 저장해 최신본 혼선 생성
