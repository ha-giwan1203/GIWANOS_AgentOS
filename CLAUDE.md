# 업무리스트 프로젝트

## 프로젝트 목적
자동차 부품 제조업(삼송 G-ERP) 업무 자동화 저장소.
생산·라인배치·정산·보고 관련 문서와 데이터를 구조화하고 반복 업무를 자동화한다.

- 98_아카이브: 완료/구버전 보관
- 99_임시수집: 미분류 임시 파일

## 상태 원본 단일화 규칙
작업의 완료/미완료/진행중/차단 상태는 **TASKS.md에만 기록한다.**
- STATUS.md: 운영 요약, 재개 위치만. 독립적 상태 판정 금지
- HANDOFF.md: 세션 변경사항과 다음 AI 액션만. 상태 선언 금지
- 판정 우선순위: TASKS.md → STATUS.md → HANDOFF.md → CLAUDE.md → Notion

## 도메인 진입 프로토콜

> `domain_guard.sh`(v2)가 도메인 활성 시 Read(대상 CLAUDE.md) 외 모든 도구를 차단한다.

| 토론 / 토론모드 / GPT토론 / 공동작업 | `90_공통기준/토론모드/CLAUDE.md` |

| 일상점검 / ZDM / 점검 입력 | `90_공통기준/스킬/zdm-daily-inspection/SKILL.md` |

## 작업 종료 시 상태 갱신
1. `TASKS.md` — 완료 이동, 신규 항목 추가
2. 도메인 `STATUS.md` — 재개 위치 기록
3. `HANDOFF.md` — 세션 변경사항·다음 AI 할 일
4. Notion — 필요 시 TASKS 기준 단방향 반영

## 기준 원본
- 기준 원본은 Git 최신 반영 결과물이다
- Git 내용과 충돌하면 Git를 우선한다
- Notion은 보조 저장소다
- 과거 메모, 업로드 파일, 대화 요약은 참고용일 뿐 기준이 아니다
- 푸시되지 않은 로컬 상태는 완료본으로 간주하지 않는다

## 공통 작업 원칙
- 기준 파일과 마스터 파일을 우선 존중한다
- 접두어, 코드, 식별값은 사용자 확인 없이 임의 제거하지 않는다
- 애매한 경우 확정하지 않고 판정유보 또는 추가검증으로 처리한다
- 반복 절차는 skill로 분리한다
- 토론모드(기존 공동작업) 검토 및 판정 기준은 `90_공통기준/토론모드/CLAUDE.md`를 우선 따른다

## 완료 판정 원칙
다음 조건이 충족될 때만 완료 또는 PASS로 판정한다.
- 기준 문서를 실제로 확인했다
- 반영 위치와 파일 경로를 확인했다
- Git 또는 Notion에 실제 결과물이 존재한다
- `TASKS.md` 상태와 충돌하지 않는다
- 설명이 아니라 실제 반영 내용을 근거로 판정했다

위 조건이 충족되지 않으면 완료 대신 아래 표현을 사용한다.
- 정합 / 부분반영 / 미반영 / 보류 / 기준 미확인

## 스킬 사용 기준
> 상세: `90_공통기준/업무관리/skill_guide.md`

- `skill-creator` — 새 스킬 설계·수정·평가·패키징 요청 시. 일반 업무 자동화 요청에는 사용하지 않는다
- `adversarial-review` — 초안·정책·구조·의사결정 비판적 검토 시. 단순 요약·설명에는 사용하지 않는다
- `line-mapping-validator` — 기준 품번과 라인배정 정합성 검증 시. 일반 품번 조회에는 사용하지 않는다
- `line-batch-management` — ERP 라인배치 입력 자동화 시. 라인배치 검증만 필요하면 사용하지 않는다
- `line-batch-mainsub` — 메인서브 품번 검색 기반 배치 시. OUTER 라인 배치에는 사용하지 않는다
- `assembly-cost-settlement` — 조립비 정산 DB·리포트 작성 시. 일반 엑셀 작업에는 사용하지 않는다
- `zdm-daily-inspection` — ZDM 일상점검 자동 입력 시. 점검 기준 문의만 있으면 사용하지 않는다
- `xlsx/docx/pdf/pptx` — 해당 파일 형식 작업 시. 단순 내용 질문에는 사용하지 않는다

## 질문 · 출력 · 보고
- 질문 최소화: 실행 가능하면 즉시 진행. 부족한 값만 1줄로 요청
- 출력: 요약 → 상세표 → 문제/원인/대응. 토론모드 검토 응답은 표 금지(문단형)
- 보고: PASS/FAIL/INFO 명시, 추정/가정/확정/검증필요 구분

## 금지사항
- 사용자 승인 없이 파일명, 시트명, 컬럼명, 코드 체계 변경
- 검증 없이 값 덮어쓰기 / 결과만 맞추기 위한 임시 보정
- 원본 파일 직접 수정 / 폴더 구조 임의 변경
- 같은 자료를 여러 저장소에 중복 저장
- 설명을 길게 늘려 토큰 낭비 / 같은 내용 반복 질문
- Git 확인 없이 추측으로 답변
- 푸시되지 않은 로컬 상태를 완료본으로 간주
- 대화 요약만으로 승인
- 토론모드 검토 응답에서 표 형식 사용
- Claude 설명만 듣고 PASS 판정
